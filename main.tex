\section{Introduction} %


This document contains the necessary documentation for the comprehesion of  the construction of a syntactic analyzer and a Syntax-Directed Translation (SDT) , which extend the lexical analyzer developed in the previous phase of this project. The system processes source code written in a subset of the C language, performing syntactic analysis through a bottom-up LALR(1) parser and semantic validation by means of SDT rules applied over the resulting Abstract Syntax Tree (AST).

The lexical phase, previously implemented using the Python Pygments library, produces a token flow that serves as input to the syntactic phase. Since PLY.YACC the tool selected for the LALR(1) parser, it requires a specific lexer interface, an adapter module was developed in order to integrate the two components. The parser then constructs an AST that is later traversed by the semantic analyzer, which enforces the typing and scoping rules of the language.

We should say that , as in the previous phase, the use of the FLEX and BISON tools is not permitted. Consequently, PLY (Python Lex-Yacc) was selected as the parser generator, which imposes specific requirements on both the grammar structure and the lexer output format, leading to a significant redesign of the Context-Free Grammar defined in the previous phase.




\subsection*{Objectives}

\begin{itemize}
    \item Extend the lexical analyzer developed in the previous phase into a full syntactic and semantic analysis for a subset of the C language.
    \item Adapt the Context-Free Grammar defined in the previous phase to be compatible with PLY.YACC's LALR(1) bottom-up parsing strategy, converting right-recursive productions to left-recursive form and delegating operator precedence to PLY's precedence mechanism.
    \item Construct an Abstract Syntax Tree (AST) during syntactic analysis by associating semantic actions with each grammar production rule, following the Syntax-Directed Translation approach.
    \item Implement a symbol table with scope management to enforce semantic rules, including variable declaration before use, type compatibility in assignments, valid operands in expressions, and correct argument usage in printf and scanf.
    \item Provide both a terminal-based and a Streamlit graphical interface for code input, analysis execution, and result visualization, including an interactive AST graph view.
\end{itemize}

\section{Theorical Framework}
Firstly, the construction of a syntactic and semantic analyzer requires a solid understanding of several theoretical concepts that form the foundation of compiler design. [1]

In this project we address the intermediate stages of the front-end of the compiler: syntactic analysis and semantic validation through Syntax-Directed Translation. These phases work together to verify that the source code provided by the user is both structurally and semantically correct before proceeding with any code generation. As in the previous phase, the validation process is applied to code written in C that uses a specific set of instructions.


\subsection*{2.1 Syntactic Analysis (Parsing) and LALR(1)}

The syntactic analyzer is the second stage of the compiler. It receives the token flow generated by the lexical analyzer and verifies that the sequence in order to acomplish the structural rules of the language, as defined by a Context-Free Grammar (CFG). [1]

In the previous phase of this project, a Top-Down Recursive Descent Parser was implemented, which constructs the parse tree starting from the initial grammar symbol and expanding downward toward the tokens. This approach required right-recursive production rules and explicit operator precedence layering within the grammar .

The current phase replaces this approach with a Bottom-Up LALR(1) parser, as implemented by PLY.YACC. In bottom-up parsing, the parse tree is constructed from the leaves (tokens) upward to the root (start symbol) through a sequence of shift and reduce operations. LALR(1) (Look-Ahead LR with 1 token of lookahead) is a widely used variant, it combines similar parser states to make a smaller and more efficient parsing table, while still working for most real programming languages.

A big advantage is that it handles left-recursive rules (which cause problems in top-down parsers) without issues in fact, they help make parsing faster.

Also, PLY.YACC lets you directly define operator precedence and associativity, so you don’t have to complicate your grammar to handle them.

\subsection*{2.2 Grammar Adaptation for LALR(1)}

A Context-Free Grammar (CFG) is a way to describe how a language is built. It is defined by four parts: G=(V,T,P,S), where V is the set of non-terminal symbols (things that can be expanded), T is the set of terminal symbols (actual tokens of the language), P is the set of production rules written like A $\rightarrow$ $\alpha$, and S is the start symbol. [3]

In the previous phase, the grammar used right-recursive rules to avoid left recursion, since top-down parsers cannot handle left recursion. For the LALR(1) parser, these rules were changed to left-recursive form, as recommended by PLY. For example:

\begin{center} 

\texttt{stmt\_list $\rightarrow$ stmt | stmt stmt\_list} \quad

\textit{(right-recursive, previous phase)}\\ 

\texttt{stmt\_list $\rightarrow$ stmt\_list stmt | stmt} \quad 

\textit{(left-recursive, current phase)} 

\end{center}

Also, operator precedence (like for arithmetic and logical operations) was removed from the grammar rules and instead defined using PLY’s precedence tuple. This lets you specify priority and associativity (left, right, or none) in a simpler way, making the grammar cleaner.

Finally, epsilon (empty) productions were used as little as possible. When they were necessary—such as in optional parts of a \texttt{for}-loop—an explicit empty rule was created using a non-terminal called \texttt{empty}, since PLY.YACC does not have a special way to write epsilon directly. [2]

\subsection*{2.3 Abstract Syntax Tree (AST)}

The main goal of the parser is to take a sequence of tokens and organize them into a tree structure called an Abstract Syntax Tree (AST). Unlike a full parse tree, the AST removes unnecessary details like parentheses and punctuation, keeping only the important parts that give meaning to the program. This makes it easier to understand how the code is structured and to check that it follows the correct syntax. [1]

In this project, each node of the AST contains four things: the \textit{kind} (the type of construct, like \texttt{"assign\_stmt"} or \texttt{"if\_stmt"}), the \textit{children} (a list of child nodes or values), the \textit{lineno} (the line number in the source code, useful for errors), and the \textit{value} (the actual text for terminal symbols). The AST is built step by step during parsing, where each grammar rule creates its corresponding node using a function called \texttt{make\_node}.[6]

\subsection*{2.4 Syntax-Directed Translation (SDT)}

Syntax-Directed Translation (SDT) is a method where grammar rules are linked with actions that give them meaning. For each rule like A $\rightarrow$ $\alpha$, some actions are executed when the parser uses that rule. These actions usually compute values (called synthesized attributes) that move from the child nodes up to the parent node. [1]

In this project, SDT is used in two steps. First, during parsing, each grammar rule creates an AST node as its result. Then, after parsing, a separate semantic analyzer goes through the AST and checks meanings by applying rules to each node. This two-step process separates building the structure from checking its correctness.

During semantic analysis, a detailed SDT trace is kept. For each node visited, it records the step number, the type of node, the line number, the action performed, and the state of the symbol table at that moment. This makes it easier to see what the program is doing and helps with debugging.[5]

\subsection*{2.5 Symbol Table and Scope Management}

A symbol table is a data structure used during compilation to store information about variables (identifiers) in the program. At least, it keeps the name of each variable and its type. It is used during semantic analysis to check that variables are declared before being used and that their types are used correctly. [1]

Scope management is handled using a stack of dictionaries, where each dictionary represents a different block of code. When the parser enters a new block (like inside an \texttt{if}, \texttt{while}, or \texttt{for}), a new scope is added to the stack. When leaving the block, that scope is removed. This makes sure that variables only exist inside the block where they were declared. When searching for a variable, the system looks from the most recent scope to the oldest one, following the usual scoping rules of C.[8]

\subsection*{2.6 Semantic Validation Rules}

Semantic analysis checks that a program makes sense, even if it is written with correct syntax. In this project, the following simple rules are applied:

\begin{itemize}
\item \textbf{Declaration before use:} A variable must be declared before it is used. If not, an “undeclared variable” error is reported with the line number.

\item \textbf{Type compatibility in assignment:} The value assigned to a variable must match its type. Types follow this order for compatibility: \texttt{char} $\le$ \texttt{int} $\le$ \texttt{float} $\le$ \texttt{double}.

\item \textbf{Valid operands in expressions:} Values used in expressions must be valid types like numbers or characters. If a variable is not declared, it causes an error that affects the whole expression.

\item \textbf{\texttt{printf} and \texttt{scanf} argument validation:} The number and types of arguments must match the format specifiers. For example, \texttt{\%d} is for \texttt{int}, \texttt{\%f} for \texttt{float}/\texttt{double}, \texttt{\%c} for \texttt{char}, and \texttt{\%s} for strings. If they don’t match, an error is generated.

\item \textbf{Evaluable conditions:} Conditions in \texttt{if}, \texttt{while}, and \texttt{for} must be of a type that can be evaluated as true or false, such as \texttt{bool}, \texttt{int}, \texttt{float}, \texttt{double}, or \texttt{char}.

\end{itemize}

\subsection*{2.7 Lexer Adapter and PLY Compatibility}

PLY.YACC expects the lexer to have a \texttt{token()} method that returns one token at a time, where each token has \texttt{.type} and \texttt{.value}. When there are no more tokens, it should return \texttt{None}. However, the lexer used in this project (based on Pygments) gives a list of \texttt{(type, value)} pairs, which is not directly compatible.[7]

To fix this, an Adapter pattern was used. The class \texttt{PlyLexerAdapter} takes the list of tokens, converts the type names to the uppercase format expected by PLY, and provides a \texttt{token()} method that returns tokens one by one.

This way, the difference between the lexer and the parser is handled in one place, so both parts can work correctly without needing to change their original design. [4]
