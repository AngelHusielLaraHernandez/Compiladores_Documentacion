\section{Introduction} %

This document contains the necessary documentation for the understanding in the construction of a lexical analyzer, where it will process source code by reading it from a file or text string. This program will be able to identify, classify and count the tokens in the code that was provided to it, in addition these will be separated into categories such as keywords, identifiers, operators, punctuators, constants and literals.

However, the lexer program meets the restriction of not making use of the FLEX tool, so a lexical analyzer was implemented using the Python Pygments library, which provides a lexer for the C language.

Now then, the lexical analyzer represents the initial phase of a compilation process, where this sets the foundations for the formulation and improvement of a context-free grammar (CFG), this will use a left factorization, which later will be used for the construction of a parser, which will be responsible for validating the syntactic structure of the code that we want to compile, ensuring that it complies with the rules defined in the CFG.

\subsection*{Objectives}

\begin{itemize}
    \item Build a lexical analyzer capable of processing code through the passing of a file or a text string.
    
    \item Generate a table with the lexical analysis of the input, classifying the tokens (such as keywords, identifiers, operators, punctuators, constants and literals) and giving a count for each category.
    
    \item Provide a basic proposal of Context-Free Grammar (CFG) that represents one of the reading examples, applying left factorization to improve that grammar.
    
    \item Package the application following software engineering practices, name it as \\
\texttt{unam.fi.compilers.g5.00} and share the repository link with public permissions.

\end{itemize}

\section{Theorical Framework}
Firstly, the traduction of a high level programming languaje into a low level one is a process structured in multiple phases. [1]

In this proyect we will aboard the inicial stages of the front-end of the compiler: the lexical analysis and the syntatic analysis. 
This phases work together in order to validate that the source code given by the user is well formed before proceding with the semantic evaluation or the code generation. In this case, the validation procees its aplied to a code that is written in C that uyses a specific set of instructions

\subsection{Lexical Analysis (Scanner / Lexer)}
The lexical analysis represents the first stage of the languaje processing. Its fundamental responsability is to redd the code character by character and group them in secuences with logical meaning,
known as lexemes.[1]

Starting from this lexemes, the analyzer, generatees a secuence of tokens (as identifiers, key words, constants, special characters, operators, punctuation, literals).
In the proyect, the lexical analyzer identifies specific sets as:
\begin{itemize}
    \item \textbf{ID:} Identifiers used to give name to the variables
    \item \textbf{VALUE\_NUM: } Literals that represent any number whole and decimal
    \item \textbf{VALUE\_CHAR: } Literals that represent a unique character enclosed between simple commas
    \item \textbf{TEXT\_STRING: } Text strings enclosed in double quotes, mostly used as format specifiers
\end{itemize}

\subsection{Syntactic analysis (Parsing) and CFG (Context-Free Grammars)}

 The syntatic analyzer is the second stage in this compiler. It recibes the flow of tokens generate by the lexical analizer and verifies that the secuence fullfill the structural rules of the language. [1]

 To define this rules in a rigurous way, it uses the CFG (CONTEXT-FREE GRAMMARS), it helps to stablish which rules are valid using the production rules.[1]
 
 In this proyect, the CFG defines the structure to prodecue any instruction (StmtList), covering numeric declarations (DeclNum), assignments (AssignSmt), and flow control statements (lfStmt, WhileStmt, ForStmtm)

 An example given is the formal rule of declaring a number (and optionally initialize it) it defines matematically in the next way:


 \vspace{0.5 cm }
 \begin{center}
     $DeclNum -> TypeNum ID AssignOpNum $
 \end{center}

 \vspace{0.5 cm }

The main objective of the parser is to organize the sequence of tokens into a hierarchical structure called a parse tree (or syntax tree). This structure shows how the tokens relate to each other according to the language rules and confirms that the code follows the correct syntax [2]

\subsection{Parsing Strategies: Top-Down Approach and Precedence Resolution}

 There are two main ways to construct the syntax tree: Top-Down and Bottom-Up. This project implements a Top-Down approach through a Recursive Descent Parser. In this method, the tree is built starting from the initial symbol of the grammar down to the leaves (the tokens), associating a recursive function with each grammar rule to process the input from left to right. [2]

To ensure that this approach works mathematically without entering infinite loops and respecting the operation hierarchies, the grammar was designed with specific features: 

\begin{itemize}
    \item \textbf{Elimination of left recursion (Left-Recursion): } The production rules for mathematical expressions (MathExpr) were rewritten to be right-recursive.
    \item \textbf{Layered Precedence:} To evaluate logical conditions (Cond) while respecting the standardized hierarchy, the rules were divided into levels. The analysis starts at the lowest priority level looking for the OR logical operator (CondOrPrime), descends to the intermediate AND level (CondAndPrime), evaluates the level of relational comparisons (CondRel), and culminates at the highest priority level processing the logical NOT and parentheses (CondUnary). 
\end{itemize}

\subsection{Grammatic}

\subsubsection*{1. Production Rules }
\begin{center}
\begin{align*}
    \text{Stmt} &\rightarrow \text{Stmt} \ \text{StmtList} \mid \varepsilon \\
    \text{StmtList} &\rightarrow \text{DeclNum} \mid \text{DeclChar} \mid \text{AssignStmt} \mid \text{IfStmt} \mid \text{WhileStmt} \\
    &\mid \text{ForStmt} \mid \text{PrintStmt} \mid \text{ScanStmt} \mid \text{UnaryStmt}
\end{align*}
\end{center}

Donde:

\begin{itemize}
    \item \textbf{Stmt} Is the initial symbol to generate instructions.
    \item \textbf{StmtList} Represents the specific instruction that can be produced.
    \item \textbf{DeclNum} y \textbf{DeclChar} Its the declarations of numbers and characters.
    \item \textbf{AssignStmt} Is for the variables asignation.
    \item \textbf{IfStmt, WhileStmt, ForStmt} they are the flow control structures.
    \item \textbf{PrintStmt, ScanStmt} they are the input and output functions of data.
    \item \textbf{UnaryStmt} it is for unary statements (pre/post increments).
\end{itemize}

\subsubsection*{2.  Production rules: }

\begin{align*}
    \text{DeclNum} &\rightarrow \text{TypeNum} \ \text{ID} \ \text{AssignOpNum} \text{;} \\
    \text{AssignOpNum} &\rightarrow \text{MathExpr} \mid \varepsilon \\
    \text{DeclChar} &\rightarrow \text{TypeChar} \ \text{ID} \ \text{AssignOpChar} \text{;} \\
    \text{AssignOpChar} &\rightarrow \text{VALUE\_CHAR} \mid \varepsilon \\
    \text{AssignStmt} &\rightarrow \text{ID} \ \text{AssignExprOp} \ \text{MathExpr} \mid \text{ID} \ \text{VALUE\_CHAR} \text{;} \\
    \text{TypeNum} &\rightarrow \text{int} \mid \text{float} \mid \text{double} \\
    \text{TypeChar} &\rightarrow \text{char}
\end{align*}

\newpage

Donde:

\begin{itemize}
    \item DeclNum : Is the PR to declare a number or declare a number and initialize it.
    \item DeclChar :  PR to declare a character or declare a character with a number and initialize it.
    \item AssignStmt : Is the PR for assigning a value to a data previously declared.
    \item TypeNum : The data type numeric that it accepts.
    \item TypeChar : Tipe of character data that accepts.
\end{itemize}


\textbf{3. Applying left factoring: }

The variable declarations $DeclNum \rightarrow TypeNum \ ID \ ; \mid TypeNum \ ID \ = \ MathExpr \ ;$ has the common factor $TypeNum \ ID$, so we can express it like:

\begin{align*}
    \text{DeclNum} &\rightarrow \text{TypeNum} \ \text{ID} \ \text{AssignOpNum} \text{;} \\
    \text{AssignOpNum} &\rightarrow \text{MathExpr} \mid \varepsilon
\end{align*}

Also the expression for continuous operations 
\begin{small}
\[
\text{MathExpr} \rightarrow \text{MathExpr} \ \text{MathOp} \ \text{MathOperand} \mid \text{MathOperand}
\]
\end{small}

presents left recursion, so it was factored (right recursion) as follows:

\begin{small}
\begin{align*}
    \text{MathExpr} &\rightarrow \text{MathOperand} \\
    &\mid \text{MathOperand} \ \text{MathOp} \ \text{MathExpr}
\end{align*}
\end{small}

This procedure will help to avoid non-determinism, also with the help of the right recursion already done, where we have non-terminal elements on the right side, so this can help with the repetitive and recursive strings that we can make when nesting operations or logical calls.

Finally the gramatic will be defined by the expressions:

\begin{small}
\begin{align*}
    \text{Stmt} &\rightarrow \text{Stmt} \ \text{StmtList} \mid \varepsilon \\
    \text{StmtList} &\rightarrow \text{DeclNum} \mid \text{DeclChar} \mid \text{AssignStmt} \mid \text{IfStmt} \mid \text{WhileStmt} \mid \\
    &\quad \text{ForStmt} \mid \text{PrintStmt} \mid \text{ScanStmt} \mid \text{UnaryStmt} \\
    \text{DeclNum} &\rightarrow \text{TypeNum} \ \text{ID} \ \text{AssignOpNum} \text{;} \\
    \text{AssignOpNum} &\rightarrow \text{MathExpr} \mid \varepsilon \\
    \text{DeclChar} &\rightarrow \text{TypeChar} \ \text{ID} \ \text{AssignOpChar} \text{;} \\
    \text{AssignOpChar} &\rightarrow \text{VALUE\_CHAR} \mid \varepsilon \\
    \text{AssignStmt} &\rightarrow \text{ID} \ \text{AssignExprOp} \ \text{MathExpr} \mid \text{ID} \ \text{VALUE\_CHAR} \text{;} \\
    \text{TypeNum} &\rightarrow \text{int} \mid \text{float} \mid \text{double} \\
    \text{TypeChar} &\rightarrow \text{char} \\
    \text{IfStmt} &\rightarrow \text{if} \ (\text{Cond}) \ \{ \text{StmtList} \} \\
    \text{WhileStmt} &\rightarrow \text{while} \ (\text{Cond}) \ \{ \text{StmtList} \} \\
    \text{ForStmt} &\rightarrow \text{for} \ (\text{ForInit} \text{;} \ \text{ForCond} \text{;} \ \text{ForStep}) \ \{ \text{StmtList} \} \\
    \text{ForInit} &\rightarrow \text{TypeNum} \ \text{ID} \ \text{AssignOpNum} \mid \text{ID} \ \text{AssignExprOp} \ \text{MathExpr} \mid \varepsilon \\
    \text{ForCond} &\rightarrow \text{Cond} \mid \varepsilon \\
    \text{ForStep} &\rightarrow \text{IncDecOp} \ \text{ID} \mid \text{ID} \ \text{IncDecOp} \mid \\
    &\quad \text{ID} \ \text{AssignExprOp} \ \text{MathExpr} \mid \varepsilon \\
    \text{PrintStmt} &\rightarrow \text{printf} ( \text{PrintArgs} ) \text{;} \\
    \text{PrintArgs} &\rightarrow \text{TEXT\_STRING} \mid \text{TEXT\_STRING} , \text{IdList} \text{;} \\
    \text{ScanStmt} &\rightarrow \text{scanf} ( \text{TEXT\_STRING} , \&\text{ID} ) \text{;} \\
    \text{IdList} &\rightarrow \text{ID} \mid \text{ID} , \text{IdList} \\
    \text{UnaryStmt} &\rightarrow \text{IncDecOp} \ \text{ID} \text{;} \mid \text{ID} \ \text{IncDecOp} \text{;} \\
    \text{IncDecOp} &\rightarrow \text{++} \mid \text{--} \\
    \text{AssignExprOp} &\rightarrow \text{=} \mid \text{+=} \mid \text{-=} \mid \text{/=} \\
    \text{MathExpr} &\rightarrow \text{MathOperand} \mid \text{MathOperand} \ \text{MathOp} \ \text{MathExpr} \\
    \text{MathOperand} &\rightarrow \text{ID} \mid \text{VALUE\_NUM} \\
    \text{MathOp} &\rightarrow + \mid - \mid * \mid / \mid \% \\
    \text{Cond} &\rightarrow \text{CondAnd} \ \text{CondOrPrime} \\
    \text{CondOrPrime} &\rightarrow || \ \text{CondAnd} \ \text{CondOrPrime} \mid \varepsilon \\
    \text{CondAnd} &\rightarrow \text{CondRel} \ \text{CondAndPrime} \\
    \text{CondAndPrime} &\rightarrow \&\& \ \text{CondRel} \ \text{CondAndPrime} \mid \varepsilon \\
    \text{CondRel} &\rightarrow \text{MathExpr} \ \text{RelOp} \ \text{MathExpr} \mid \text{CondUnary} \\
    \text{CondUnary} &\rightarrow \text{!} \ \text{CondUnary} \mid (\text{Cond}) \mid \text{IdVal} \\
    \text{IdVal} &\rightarrow \text{ID} \mid \text{VALUE\_NUM} \mid \text{VALUE\_CHAR} \\
    \text{RelOp} &\rightarrow == \mid != \mid < \mid > \mid <= \mid >=
\end{align*}
\end{small}

\newpage

\textbf{4. Logical conditions (Layer precedence)} 

For rigorously respecting the hierarchy of logical operations, the production rules for conditions (\texttt{Cond}) were designed using a stratified priority level approach:
\begin{itemize}
    \item \textbf{Nivel de Menor Prioridad (OR):}  The analysis starts in \texttt{Cond}, obligating the evaluation of lower layers before processing the \texttt{||} operator.
    \item \textbf{Nivel Intermedio (AND):} Through \texttt{CondAnd}, the operators group their logical operands before the upper layer (OR) can process them.
    \item \textbf{Nivel de Comparaciones (Relacional):} The rule \texttt{CondRel} evaluates the mathematical expressions separated by relational operators ($>$, $<$, $==$).
    \item \textbf{Nivel de Máxima Prioridad (NOT y Paréntesis):} The rule \texttt{CondUnary} ensures that logical negations and nested expressions between parentheses are resolved first.
\end{itemize}

\textbf{Formal Description of the Automaton:}

\vspace{0.3cm}

The deterministic finite automaton is formally defined as the 5-tuple $M = (Q, \Sigma, \delta, q_0, F)$ where:

\begin{itemize}
    \item $Q = \{q_0, q_1, q_2, \ldots, q_{46}\}$ \quad (47 states)
    \item $\Sigma$ is the set of input symbols (terminals and non-terminals of the grammar)
    \item $\delta : Q \times \Sigma \rightarrow Q$ is the transition function defined below
    \item $q_0$ is the initial state
    \item $F = \{q_0\}$ is the set of accepting states
\end{itemize}



\vspace{0.3cm}
\textbf{Observations about the automaton:}

\begin{enumerate}
    \item \textbf{State $q_0$ as initial and accepting state:} The state $q_0$ fulfills both roles because the production $\text{Stmt} \rightarrow \varepsilon$ allows accepting the empty string (a program without statements). After completely recognizing a statement, the automaton always returns to $q_0$ to initiate the recognition of the next one.

    \item \textbf{Determinism by anticipation token (\textit{lookahead}):} From $q_0$, the first token of the input determines unequivocally the path to follow:
    \begin{itemize}
        \item \texttt{int}, \texttt{float}, \texttt{double} $\rightarrow$ DeclNum (state $q_1$)
        \item \texttt{char} $\rightarrow$ DeclChar (state $q_5$)
        \item \texttt{ID} $\rightarrow$ AssignStmt o UnaryStmt posfijo (state $q_9$)
        \item \texttt{++}, \texttt{-{}-} $\rightarrow$ UnaryStmt prefijo (state $q_{12}$)
        \item \texttt{if} $\rightarrow$ IfStmt (state $q_{14}$)
        \item \texttt{while} $\rightarrow$ WhileStmt (state $q_{20}$)
        \item \texttt{for} $\rightarrow$ ForStmt (state $q_{26}$)
        \item \texttt{printf} $\rightarrow$ PrintStmt (state $q_{36}$)
        \item \texttt{scanf} $\rightarrow$ ScanStmt (state $q_{40}$)
    \end{itemize}

    \item \textbf{Optional transitions($\varepsilon$):} The states $q_2$ y $q_6$ posses two transitions for \texttt{;}: uone direct to $q_0$ (when the variable is declared without initialization, it means $\text{AssignOpNum} \rightarrow \varepsilon$) and another that passes through the assignment before reaching the final \texttt{;}.

    \item \textbf{Ambiguety reolved in $q_9$:} When the first token is \texttt{ID}, the automaton transitions to $q_9$. From there, the second token resolves the ambiguity:
    \begin{itemize}
        \item If it is an \textit{AssignExprOp} (\texttt{=}, \texttt{+=}, \texttt{-=}, \texttt{/=}) $\rightarrow$ the path of AssignStmt is followed.
        \item If it is an \textit{IncDecOp} (\texttt{++}, \texttt{-{}-}) $\rightarrow$ the path of UnaryStmt posfijo is followed.
    \end{itemize}

    \item \textbf{Non-terminal symbols as transitions:} The labels like \textit{Cond}, \textit{StmtList}, \textit{MathExpr}, \textit{PrintArgs}, \textit{ForInit}, \textit{ForCond} y \textit{ForStep} represent recursive calls to the predictive parser, where each non-terminal expands according to its own production rules defined in the grammar.
\end{enumerate}

\subsection{Input/Output and execution}

The lexical analyzer developed supports two modes of operation for C language inputs: reading from a file and reading from the keyboard.

\begin{enumerate}
    \item \textbf{Read from file.}  
    In this mode, the user must place the file in the inputs folder and the system reads it using UTF-8 encoding and subsequently processes the content line by line. At the end of the lexical analysis, the results can be exported to the outputs folder.

    \item \textbf{Read from keyboard.}  
    The analyzer allows entering source code directly from the terminal. The capture ends when the user inputs the terminal symbol \texttt{\$}, which acts as a delimiter for the end of input.  
 \end{enumerate}

During execution, the system performs the following main operations:

\begin{itemize}
    \item Tokenization of the source code using the C lexer provided by the \textit{Pygments} library.
    \item Classification of each token according to the categories defined in the course (Keyword, Identifier, Operator, Punctuator, Constant and Literal).
    \item Removal of irrelevant tokens such as whitespace or line breaks that do not provide syntactic information.
    \item Grouping and total counting of tokens by category.
    \item Optional visualization of the source code with token highlighting in colors and exportation of results to a text file.
\end{itemize}

It is worth mentioning that the particular case of the character \texttt{\&} used in the \texttt{scanf()} function, as well as certain delimiters such as parentheses, braces, commas and semicolons were explicitly treated as punctuators because the library used does not categorize them consistently.

\subsection{Relation with the propose CFG}

The design of the lexical analyzer was carried out taking as reference the context-free grammar defined for the reduced C language considered in this project.
Our CFG includes instructions such as declarations of numeric and character variables, assignments, control structures (\texttt{if}, \texttt{while}, \texttt{for}), unary increment and decrement operations, as well as calls to input/output functions (\texttt{printf} and \texttt{scanf}) \textit{\cite{cfglexer}}. 

Also we defined production rules for mathematical expressions and logical conditions respecting operator precedence through a layered derivation approach, where logical OR operators have the lowest priority, followed by AND, relational operators and finally unary operators such as logical negation and the use of parentheses.

\subsection{Tests realized}

In order to validate our lexical analyzer, a couple of tests were proposed using fragments of code in the C language that include declarations, arithmetic expressions, calls to input/output functions and control structures contemplated in the proposed CFG.
\begin{itemize}
    \item \textbf{Code 1: Statements and arithmetic expressions}  

    In this test, a program was analyzed that includes declarations of numeric variables, an assignment with an arithmetic expression and a call to the \texttt{printf()} function. The lexical analyzer correctly identified keywords, identifiers, assignment operators and arithmetic operators, as well as numeric constants and language delimiters.
    
    Similarly, the proper recognition of the string literal used in the output function was verified, which was grouped as a single token. The result of the analysis made it possible to obtain the total count of tokens classified by category, validating the correct tokenization of arithmetic expressions and print statements.
    \item \textbf{Code 2: Data input and iterative structure}  

    In this test, a program was evaluated that performs data reading using \texttt{scanf()} and employs an iterative structure \texttt{for}. During the analysis, the associated keywords for the loop, relational and increment operators were correctly recognized, as well as the corresponding delimiters.

    Additionally, the classification of the special symbol \texttt{\&} used for reference reading and the correct identification of the string literal within the \texttt{printf()} function were verified. The results confirmed that the lexical analyzer can adequately process iterative statements and calls to input/output functions, generating the table and grouped count of tokens.
\end{itemize}
