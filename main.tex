\section{Introduction} %


This document contains the necessary documentation for the comprehesion of  the construction of a syntactic analyzer and a Syntax-Directed Translation (SDT) , which extend the lexical analyzer developed in the previous phase of this project. The system processes source code written in a subset of the C language, performing syntactic analysis through a bottom-up LALR(1) parser and semantic validation by means of SDT rules applied over the resulting Abstract Syntax Tree (AST).

The lexical phase, previously implemented using the Python Pygments library, produces a token flow that serves as input to the syntactic phase. Since PLY.YACC the tool selected for the LALR(1) parser, it requires a specific lexer interface, an adapter module was developed in order to integrate the two components. The parser then constructs an AST that is later traversed by the semantic analyzer, which enforces the typing and scoping rules of the language.

We should say that , as in the previous phase, the use of the FLEX and BISON tools is not permitted. Consequently, PLY (Python Lex-Yacc) was selected as the parser generator, which imposes specific requirements on both the grammar structure and the lexer output format, leading to a significant redesign of the Context-Free Grammar defined in the previous phase.




\subsection*{Objectives}

\begin{itemize}
    \item Extend the lexical analyzer developed in the previous phase into a full syntactic and semantic analysis for a subset of the C language.
    \item Adapt the Context-Free Grammar defined in the previous phase to be compatible with PLY.YACC's LALR(1) bottom-up parsing strategy, converting right-recursive productions to left-recursive form and delegating operator precedence to PLY's precedence mechanism.
    \item Construct an Abstract Syntax Tree (AST) during syntactic analysis by associating semantic actions with each grammar production rule, following the Syntax-Directed Translation approach.
    \item Implement a symbol table with scope management to enforce semantic rules, including variable declaration before use, type compatibility in assignments, valid operands in expressions, and correct argument usage in printf and scanf.
    \item Provide both a terminal-based and a Streamlit graphical interface for code input, analysis execution, and result visualization, including an interactive AST graph view.
\end{itemize}

\section{Theorical Framework}
Firstly, the construction of a syntactic and semantic analyzer requires a solid understanding of several theoretical concepts that form the foundation of compiler design. [1]

In this project we address the intermediate stages of the front-end of the compiler: syntactic analysis and semantic validation through Syntax-Directed Translation. These phases work together to verify that the source code provided by the user is both structurally and semantically correct before proceeding with any code generation. As in the previous phase, the validation process is applied to code written in C that uses a specific set of instructions.


\subsection*{2.1 Syntactic Analysis (Parsing) and LALR(1)}

The syntactic analyzer is the second stage of the compiler. It receives the token flow generated by the lexical analyzer and verifies that the sequence in order to acomplish the structural rules of the language, as defined by a Context-Free Grammar (CFG). [1]

In the previous phase of this project, a Top-Down Recursive Descent Parser was implemented, which constructs the parse tree starting from the initial grammar symbol and expanding downward toward the tokens. This approach required right-recursive production rules and explicit operator precedence layering within the grammar .

The current phase replaces this approach with a Bottom-Up LALR(1) parser, as implemented by PLY.YACC. In bottom-up parsing, the parse tree is constructed from the leaves (tokens) upward to the root (start symbol) through a sequence of shift and reduce operations. LALR(1) (Look-Ahead LR with 1 token of lookahead) is a widely used variant, it combines similar parser states to make a smaller and more efficient parsing table, while still working for most real programming languages.

A big advantage is that it handles left-recursive rules (which cause problems in top-down parsers) without issues in fact, they help make parsing faster.

Also, PLY.YACC lets you directly define operator precedence and associativity, so you don’t have to complicate your grammar to handle them.

\subsection*{2.2 Grammar Adaptation for LALR(1)}

A Context-Free Grammar (CFG) is a way to describe how a language is built. It is defined by four parts: G=(V,T,P,S), where V is the set of non-terminal symbols (things that can be expanded), T is the set of terminal symbols (actual tokens of the language), P is the set of production rules written like A $\rightarrow$ $\alpha$, and S is the start symbol. [3]

In the previous phase, the grammar used right-recursive rules to avoid left recursion, since top-down parsers cannot handle left recursion. For the LALR(1) parser, these rules were changed to left-recursive form, as recommended by PLY. For example:

\begin{center} 

\texttt{stmt\_list $\rightarrow$ stmt | stmt stmt\_list} \quad

\textit{(right-recursive, previous phase)}\\ 

\texttt{stmt\_list $\rightarrow$ stmt\_list stmt | stmt} \quad 

\textit{(left-recursive, current phase)} 

\end{center}

Also, operator precedence (like for arithmetic and logical operations) was removed from the grammar rules and instead defined using PLY’s precedence tuple. This lets you specify priority and associativity (left, right, or none) in a simpler way, making the grammar cleaner.

Finally, epsilon (empty) productions were used as little as possible. When they were necessary—such as in optional parts of a \texttt{for}-loop—an explicit empty rule was created using a non-terminal called \texttt{empty}, since PLY.YACC does not have a special way to write epsilon directly. [2]

\subsection*{2.3 Abstract Syntax Tree (AST)}

The main goal of the parser is to take a sequence of tokens and organize them into a tree structure called an Abstract Syntax Tree (AST). Unlike a full parse tree, the AST removes unnecessary details like parentheses and punctuation, keeping only the important parts that give meaning to the program. This makes it easier to understand how the code is structured and to check that it follows the correct syntax. [1]

In this project, each node of the AST contains four things: the \textit{kind} (the type of construct, like \texttt{"assign\_stmt"} or \texttt{"if\_stmt"}), the \textit{children} (a list of child nodes or values), the \textit{lineno} (the line number in the source code, useful for errors), and the \textit{value} (the actual text for terminal symbols). The AST is built step by step during parsing, where each grammar rule creates its corresponding node using a function called \texttt{make\_node}.[6]

\subsection*{2.4 Syntax-Directed Translation (SDT)}

Syntax-Directed Translation (SDT) is a method where grammar rules are linked with actions that give them meaning. For each rule like A $\rightarrow$ $\alpha$, some actions are executed when the parser uses that rule. These actions usually compute values (called synthesized attributes) that move from the child nodes up to the parent node. [1]

In this project, SDT is used in two steps. First, during parsing, each grammar rule creates an AST node as its result. Then, after parsing, a separate semantic analyzer goes through the AST and checks meanings by applying rules to each node. This two-step process separates building the structure from checking its correctness.

During semantic analysis, a detailed SDT trace is kept. For each node visited, it records the step number, the type of node, the line number, the action performed, and the state of the symbol table at that moment. This makes it easier to see what the program is doing and helps with debugging.[5]

\subsection*{2.5 Symbol Table and Scope Management}

A symbol table is a data structure used during compilation to store information about variables (identifiers) in the program. At least, it keeps the name of each variable and its type. It is used during semantic analysis to check that variables are declared before being used and that their types are used correctly. [1]

Scope management is handled using a stack of dictionaries, where each dictionary represents a different block of code. When the parser enters a new block (like inside an \texttt{if}, \texttt{while}, or \texttt{for}), a new scope is added to the stack. When leaving the block, that scope is removed. This makes sure that variables only exist inside the block where they were declared. When searching for a variable, the system looks from the most recent scope to the oldest one, following the usual scoping rules of C.[8]

\subsection*{2.6 Semantic Validation Rules}

Semantic analysis checks that a program makes sense, even if it is written with correct syntax. In this project, the following simple rules are applied:

\begin{itemize}
\item \textbf{Declaration before use:} A variable must be declared before it is used. If not, an “undeclared variable” error is reported with the line number.

\item \textbf{Type compatibility in assignment:} The value assigned to a variable must match its type. Types follow this order for compatibility: \texttt{char} $\le$ \texttt{int} $\le$ \texttt{float} $\le$ \texttt{double}.

\item \textbf{Valid operands in expressions:} Values used in expressions must be valid types like numbers or characters. If a variable is not declared, it causes an error that affects the whole expression.

\item \textbf{\texttt{printf} and \texttt{scanf} argument validation:} The number and types of arguments must match the format specifiers. For example, \texttt{\%d} is for \texttt{int}, \texttt{\%f} for \texttt{float}/\texttt{double}, \texttt{\%c} for \texttt{char}, and \texttt{\%s} for strings. If they don’t match, an error is generated.

\item \textbf{Evaluable conditions:} Conditions in \texttt{if}, \texttt{while}, and \texttt{for} must be of a type that can be evaluated as true or false, such as \texttt{bool}, \texttt{int}, \texttt{float}, \texttt{double}, or \texttt{char}.

\end{itemize}

\subsection*{2.7 Lexer Adapter and PLY Compatibility}

PLY.YACC expects the lexer to have a \texttt{token()} method that returns one token at a time, where each token has \texttt{.type} and \texttt{.value}. When there are no more tokens, it should return \texttt{None}. However, the lexer used in this project (based on Pygments) gives a list of \texttt{(type, value)} pairs, which is not directly compatible.[7]

To fix this, an Adapter pattern was used. The class \texttt{PlyLexerAdapter} takes the list of tokens, converts the type names to the uppercase format expected by PLY, and provides a \texttt{token()} method that returns tokens one by one.

This way, the difference between the lexer and the parser is handled in one place, so both parts can work correctly without needing to change their original design. [4]


\section{Development}

\subsection{Implementación}

El compilador siguio una arquitectura dividida en tres capas, donde de primeras tenemos 
la capa lógica, en seguida tenemos la capa de presentación que es el frontend y por ultimo tenemos a 
la capa de utilidades, para esto la capa lógica se compone de seis secciones, la primera es el analizador léxico, 
el adaptador de tokens, las reglas gramaticales, las estructuras del AST, las reglas de validación semántica 
y la fase principal que integra la logica por completo de análisis sintáctico y semántico.

El flujo  sigue un pipeline se forma secuencial, donde primero, el analizador léxico 
tokeniza el código fuente en C utilizando Pygments como base. 
Despues de esto, un bloque secundario transforma la salida del lexer al formato requerido por PLY 
(Python Lex-Yacc) y despues, el parser LALR(1) de PLY analiza los tokens 
y construye el AST. Por ultimo, el módulo semántico pasa el AST aplicando las reglas de validación definidas.

\subsubsection{Justificacion de las producciones}

La gramática se compone de 25 no terminales 
y 60 producciones que definen el lenguaje:
\vspace{0.5cm}

\textbf{Estructura del programa:} La Regla 1 define que un programa consiste en una lista de sentencias, la cual se construye de forma recursiva por la izquierda mediante las Reglas 2 y 3.

\textbf{Sentencias:} De las Reglas de la 4 a la 12 establecen que cada sentencia puede ser una de nueve alternativas: declaración numérica, declaración de carácter, asignación, condicional \texttt{if}, ciclo \texttt{while}, ciclo \texttt{for}, impresión con \texttt{printf}, lectura con \texttt{scanf}.

\textbf{Declaraciones:} De las Reglas de la 13 a la 14 definen declaraciones de variables numéricas (\texttt{int}, \texttt{float}, \texttt{double}), tanto simples como con inicialización. Para la 15 y 16 hacen lo mismo pero para el tipo carácter (Regla 18), donde la inicialización solo acepta literales de carácter.

\textbf{Asignaciones:} La Regla 19 define la asignación con operadores simples o compuestos (Regla 21: \texttt{=}, \texttt{+=}, \texttt{-=}, \texttt{/=}) sobre la parte aritmetica y la Regla 20 permite la asignación directa de letras.

\textbf{Estructuras de control:} Las Reglas 22 y 23 definen los bloques de control, como son el \texttt{if} y \texttt{while}, que requieren una condición entre paréntesis y un bloque que inicia y finaliza por medio de llaves. La Regla 24 define el ciclo \texttt{for} con tres partes separados por punto y coma: inicialización (Reglas 25--28), condición (Reglas 29--30) y paso (Reglas 31--34), cada uno de los cuales puede ser vacío ($\varepsilon$).

\textbf{Entrada/Salida:} La Regla 35 indica que lleva la instrucción \texttt{printf}, cuyos argumentos se especifican en las Reglas 36--37. La Regla 38 indica que lleva \texttt{scanf}, que acepta una cadena de formato y un identificador donde tiene que seguir un \texttt{\&}. 

\textbf{Expresiones aritméticas:} Las Reglas de la 44 a la 51 indican las expresiones matemáticas con cinco operadores, identificadores, constantes numéricas y agrupación con paréntesis. 

\textbf{Condiciones:} Las Reglas de la 52 a la 58 definen las condiciones con operadores lógicos, negación, comparaciones relacionales (Regla 59), agrupación con paréntesis y la conversión de expresiones aritméticas en booleanos.

El proceso de funcionamiento del bottom-up para esta entrada es de la siguiente manera, donde primero
el analizador desplaza el token \texttt{x} hacia la pila, despues desplaza el token \texttt{=}
y lo reduce al no terminal haciendo uso de la Regla 21. 
Despues desplaza el segundo token \texttt{x} y lo reduce a \textit{math\_expr} por medio de la Regla 49. 
Luego desplaza a la \texttt{+} y al token \texttt{1}, el cual se reduce a \textit{math\_expr} por la Regla 50.

En este punto, el parser tiene en la cima de la pila los elementos:

$$math\_expr \; \texttt{+} \; math\_expr$$ 

Que es lo mismo que la parte del lado derecho de la Regla 44.

\newpage

Con esto dicho, se aplica esta reducción lo cual genera un nodo \textit{math\_expr} 
que representa la expresión $x + 1$. Por ultimo se desplaza el token \texttt{;} 
y la pila contiene los elementos 

$$\texttt{IDENTIFIER} \; assign\_expr\_op \; math\_expr \; \texttt{;}$$ 

Y esto es lo mismo que con la Regla 19 y se aplica esta reducción creando el nodo \textit{assign\_stmt}, 
el cual se reduce a \textit{stmt} por la Regla 6, luego a \textit{stmt\_list} por la Regla 3, 
y finalmente al símbolo inicial \textit{program} mediante la Regla 1.

\subsection{Traducción Dirigida por la Sintaxis (SDT)}

La Traducción Dirigida por el (SDT \textit{Syntax-Directed Translation}) es que esta enlaza las reglas semánticas a cada producción de la gramática. 

Para esto, con la forma de trabajo ya establecida, podemos implementar y presentar 
las reglas de SDT para cada producción, donde:

$SymTable$ representa la tabla de símbolos. 

\textit{.type} es el atributo de tipo inferido.

\textit{.node} es el nodo AST generado.

\subsubsection{Reglas SDT -- Estructura del programa}

\begin{align*}
program &\rightarrow stmt\_list \quad \{program.node = \text{make\_node}(\text{``program''}, stmt\_list.nodes)\} \\[8pt]
stmt\_list &\rightarrow stmt\_list_1 \; stmt \quad \{stmt\_list.nodes = stmt\_list_1.nodes \cup \{stmt.node\}\} \\[4pt]
stmt\_list &\rightarrow stmt \quad \{stmt\_list.nodes = \{stmt.node\}\} \\[8pt]
stmt &\rightarrow decl\_num \mid decl\_char \mid assign\_stmt \mid if\_stmt \\
     &\quad\mid while\_stmt \mid for\_stmt \mid print\_stmt \mid scan\_stmt \\
     &\quad\mid unary\_stmt \quad \{stmt.node = X.node\}
\end{align*}

\subsubsection{Reglas SDT -- Declaraciones}

\begin{align*}
decl\_num &\rightarrow type\_num \; \mathbf{id} \; \mathbf{;} \\
         &\quad \{\textbf{if } SymTable.\text{declare}(\mathbf{id}.name, type\_num.type) \textbf{ then } ok \\
         &\quad \textbf{ else } \text{error}(\text{``redeclaración de variable''})\} \\[10pt]
decl\_num &\rightarrow type\_num \; \mathbf{id} \; \mathbf{=} \; math\_expr \; \mathbf{;} \\
         &\quad \{\textbf{if } SymTable.\text{declare}(\mathbf{id}.name, type\_num.type) \\
         &\quad \textbf{ then if } \text{compatible}(type\_num.type, math\_expr.type) \textbf{ then } ok \\
         &\quad \textbf{ else } \text{error}(\text{``tipo incompatible en asignación''}) \\
         &\quad \textbf{ else } \text{error}(\text{``redeclaración de variable''})\} \\[10pt]
decl\_char &\rightarrow type\_char \; \mathbf{id} \; \mathbf{;} \\
          &\quad \{\textbf{if } SymTable.\text{declare}(\mathbf{id}.name, \text{``char''}) \textbf{ then } ok \\
          &\quad \textbf{ else } \text{error}(\text{``redeclaración''})\} \\[10pt]
decl\_char &\rightarrow type\_char \; \mathbf{id} \; \mathbf{=} \; \mathbf{VALUE\_CHAR} \; \mathbf{;} \\
          &\quad \{\textbf{if } SymTable.\text{declare}(\mathbf{id}.name, \text{``char''}) \textbf{ then } ok \\
          &\quad \textbf{ else } \text{error}(\text{``redeclaración''})\} \\[10pt]
type\_num &\rightarrow \mathbf{int} \quad \{type\_num.type = \text{``int''}\} \\
type\_num &\rightarrow \mathbf{float} \quad \{type\_num.type = \text{``float''}\} \\
type\_num &\rightarrow \mathbf{double} \quad \{type\_num.type = \text{``double''}\} \\[6pt]
type\_char &\rightarrow \mathbf{char} \quad \{type\_char.type = \text{``char''}\}
\end{align*}

\subsubsection{Reglas SDT -- Asignación}

\begin{align*}
assign\_stmt &\rightarrow \mathbf{id} \; assign\_op \; math\_expr \; \mathbf{;} \\
            &\quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{null} \\
            &\quad \textbf{ then } \text{error}(\text{``variable no declarada''}) \\
            &\quad \textbf{ else if } assign\_op.op = \text{``=''} \\
            &\quad\quad \textbf{ then if } \text{compatible}(SymTable.\text{lookup}(\mathbf{id}.name), math\_expr.type) \\
            &\quad\quad \textbf{ then } ok \textbf{ else } \text{error}(\text{``tipo incompatible''}) \\
            &\quad \textbf{ else if } assign\_op.op \in \{\text{``+=''}, \text{``-=''}, \text{``/=''}\} \\
            &\quad\quad \textbf{ then if } \text{numeric}(SymTable.\text{lookup}(\mathbf{id}.name)) \land \text{numeric}(math\_expr.type) \\
            &\quad\quad \textbf{ then } ok \textbf{ else } \text{error}(\text{``operandos no numéricos''})\} \\[10pt]
assign\_stmt &\rightarrow \mathbf{id} \; \mathbf{=} \; \mathbf{VALUE\_CHAR} \; \mathbf{;} \\
            &\quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{null} \\
            &\quad \textbf{ then } \text{error}(\text{``variable no declarada''}) \\
            &\quad \textbf{ else if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{``char''} \textbf{ then } ok \\
            &\quad \textbf{ else } \text{error}(\text{``tipo incompatible''})\} \\[10pt]
assign\_op &\rightarrow \mathbf{=} \;\{assign\_op.op = \text{``=''}\} \\
           &\mid \mathbf{+=} \;\{assign\_op.op = \text{``+=''}\} \\
           &\mid \mathbf{-=} \;\{assign\_op.op = \text{``-=''}\} \\
           &\mid \mathbf{/=} \;\{assign\_op.op = \text{``/=''}\}
\end{align*}

\subsubsection{Reglas SDT -- Expresiones aritméticas}

\begin{align*}
math\_expr &\rightarrow math\_expr_1 \; \mathbf{+} \; math\_expr_2 \\
           &\quad \{\textbf{if } \text{numeric}(math\_expr_1.type) \land \text{numeric}(math\_expr_2.type) \\
           &\quad \textbf{ then } math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \\
           &\quad \textbf{ else } \text{error}(\text{``operandos no numéricos''})\} \\[6pt]
math\_expr &\rightarrow math\_expr_1 \; \mathbf{-} \; math\_expr_2 \\
           &\quad \{math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \text{ (misma regla)}\} \\[6pt]
math\_expr &\rightarrow math\_expr_1 \; \mathbf{*} \; math\_expr_2 \\
           &\quad \{math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \text{ (misma regla)}\} \\[6pt]
math\_expr &\rightarrow math\_expr_1 \; \mathbf{/} \; math\_expr_2 \\
           &\quad \{math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \text{ (misma regla)}\} \\[6pt]
math\_expr &\rightarrow math\_expr_1 \; \mathbf{\%} \; math\_expr_2 \\
           &\quad \{math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \text{ (misma regla)}\} \\[10pt]
math\_expr &\rightarrow \mathbf{id} \quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) \neq \text{null} \\
           &\quad \textbf{ then } math\_expr.type = SymTable.\text{lookup}(\mathbf{id}.name) \\
           &\quad \textbf{ else } \text{error}(\text{``variable no declarada''})\} \\[6pt]
math\_expr &\rightarrow \mathbf{VALUE\_NUM} \quad \{math\_expr.type = \text{infer}(\mathbf{VALUE\_NUM}.value)\} \\[6pt]
math\_expr &\rightarrow \mathbf{(} \; math\_expr_1 \; \mathbf{)} \quad \{math\_expr.type = math\_expr_1.type\}
\end{align*}

\subsubsection{Reglas SDT -- Condiciones}

\begin{align*}
cond &\rightarrow cond_1 \; \mathbf{||} \; cond_2 \\
     &\quad \{\textbf{if } \text{logical}(cond_1.type) \land \text{logical}(cond_2.type) \\
     &\quad \textbf{ then } cond.type = \text{``bool''} \\
     &\quad \textbf{ else } \text{error}(\text{``operandos no válidos para OR lógico''})\} \\[6pt]
cond &\rightarrow cond_1 \; \mathbf{\&\&} \; cond_2 \\
     &\quad \{\textbf{if } \text{logical}(cond_1.type) \land \text{logical}(cond_2.type) \\
     &\quad \textbf{ then } cond.type = \text{``bool''} \\
     &\quad \textbf{ else } \text{error}(\text{``operandos no válidos para AND lógico''})\} \\[6pt]
cond &\rightarrow math\_expr_1 \; rel\_op \; math\_expr_2 \\
     &\quad \{\textbf{if } \text{numeric}(math\_expr_1.type) \land \text{numeric}(math\_expr_2.type) \\
     &\quad \textbf{ then } cond.type = \text{``bool''} \\
     &\quad \textbf{ else } \text{error}(\text{``operandos no numéricos en comparación''})\} \\[6pt]
cond &\rightarrow \mathbf{!} \; cond_1 \\
     &\quad \{\textbf{if } \text{logical}(cond_1.type) \textbf{ then } cond.type = \text{``bool''} \\
     &\quad \textbf{ else } \text{error}(\text{``operando no válido para negación''})\} \\[6pt]
cond &\rightarrow \mathbf{(} \; cond_1 \; \mathbf{)} \quad \{cond.type = cond_1.type\} \\[6pt]
cond &\rightarrow math\_expr \quad \{cond.type = math\_expr.type\} \\[6pt]
cond &\rightarrow \mathbf{VALUE\_CHAR} \quad \{cond.type = \text{``char''}\}
\end{align*}

Donde la función $\text{logical}(t)$ devuelve verdadero si $t \in \{\text{``bool'', ``int'', ``float'', ``double'', ``char''}\}$, son los tipos de dato que se evaluan como 
funcion logica.

\begin{align*}
rel\_op &\rightarrow \mathbf{==} \;\{rel\_op.op = \text{``==''}\} \;\mid\; \mathbf{!=} \;\{rel\_op.op = \text{``!=''}\} \\
        &\;\mid\; \mathbf{<} \;\{rel\_op.op = \text{``<''}\} \;\mid\; \mathbf{>} \;\{rel\_op.op = \text{``>''}\} \\
        &\;\mid\; \mathbf{<=} \;\{rel\_op.op = \text{``<=''}\} \;\mid\; \mathbf{>=} \;\{rel\_op.op = \text{``>=''}\}
\end{align*}

\subsubsection{Reglas SDT -- Estructuras de control}

\begin{align*}
if\_stmt &\rightarrow \mathbf{if} \; \mathbf{(} \; cond \; \mathbf{)} \; \mathbf{\{} \; stmt\_list \; \mathbf{\}} \\
        &\quad \{\textbf{if } \text{logical}(cond.type) \textbf{ then } ok \\
        &\quad \textbf{ else } \text{error}(\text{``condición no evaluable en if''})\} \\[10pt]
while\_stmt &\rightarrow \mathbf{while} \; \mathbf{(} \; cond \; \mathbf{)} \; \mathbf{\{} \; stmt\_list \; \mathbf{\}} \\
           &\quad \{\textbf{if } \text{logical}(cond.type) \textbf{ then } ok \\
           &\quad \textbf{ else } \text{error}(\text{``condición no evaluable en while''})\} \\[10pt]
for\_stmt &\rightarrow \mathbf{for} \; \mathbf{(} \; for\_init \; \mathbf{;} \; for\_cond \; \mathbf{;} \; for\_step \; \mathbf{)} \; \mathbf{\{} \; stmt\_list \; \mathbf{\}} \\
         &\quad \{SymTable.\text{enter\_scope}(); \\
         &\quad \text{evaluar}(for\_init); \\
         &\quad \textbf{if } for\_cond \neq \varepsilon \land \neg\text{logical}(for\_cond.type) \\
         &\quad \textbf{ then } \text{error}(\text{``condición no evaluable en for''}); \\
         &\quad \text{evaluar}(for\_step); \; \text{evaluar}(stmt\_list); \\
         &\quad SymTable.\text{exit\_scope}()\}
\end{align*}

\subsubsection{Reglas SDT -- Entrada/Salida}

\begin{align*}
print\_stmt &\rightarrow \mathbf{printf} \; \mathbf{(} \; print\_args \; \mathbf{)} \; \mathbf{;} \\
           &\quad \{\text{sea } specs = \text{extract\_specs}(\mathbf{TEXT\_STRING}.value) \\
           &\quad \textbf{if } |specs| \neq |id\_list| \textbf{ then } \text{error}(\text{``cantidad de argumentos incorrecta''}) \\
           &\quad \textbf{else } \forall (spec_i, id_i) \in \text{zip}(specs, id\_list): \\
           &\quad\quad \textbf{if } SymTable.\text{lookup}(id_i) = \text{null} \\
           &\quad\quad \textbf{ then } \text{error}(\text{``variable no declarada''}) \\
           &\quad\quad \textbf{ else if } \neg\text{format\_ok}(SymTable.\text{lookup}(id_i), spec_i) \\
           &\quad\quad \textbf{ then } \text{error}(\text{``formato incompatible''})\} \\[10pt]
scan\_stmt &\rightarrow \mathbf{scanf} \; \mathbf{(} \; \mathbf{TEXT\_STRING} \; \mathbf{,} \; \mathbf{\&} \; \mathbf{id} \; \mathbf{)} \; \mathbf{;} \\
          &\quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{null} \\
          &\quad \textbf{ then } \text{error}(\text{``variable no declarada''}) \\
          &\quad \textbf{ else } \text{sea } specs = \text{extract\_specs}(\mathbf{TEXT\_STRING}.value) \\
          &\quad \textbf{if } |specs| \neq 1 \textbf{ then } \text{error}(\text{``se espera exactamente un especificador''}) \\
          &\quad \textbf{ else if } \neg\text{format\_ok}(SymTable.\text{lookup}(\mathbf{id}.name), specs[0]) \\
          &\quad \textbf{ then } \text{error}(\text{``formato incompatible con tipo de variable''})\}
\end{align*}

Donde la función $\text{format\_ok}$ verifica que sean compatibles entre un tipo de variable y un especificador de formato:
\texttt{\%d}/\texttt{\%i} necesita a una variable \texttt{int}, \texttt{\%f} necesita a una variable \texttt{float}, por ponerlos de ejemplo.

\subsubsection{Reglas SDT -- Sentencias unarias}

\begin{align*}
unary\_stmt &\rightarrow inc\_dec\_op \; \mathbf{id} \; \mathbf{;} \\
           &\quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{null} \\
           &\quad \textbf{ then } \text{error}(\text{``variable no declarada''}) \\
           &\quad \textbf{ else if } \neg\text{numeric}(SymTable.\text{lookup}(\mathbf{id}.name)) \\
           &\quad \textbf{ then } \text{error}(\text{``operador unario requiere tipo numérico''})\} \\[6pt]
unary\_stmt &\rightarrow \mathbf{id} \; inc\_dec\_op \; \mathbf{;} \quad \text{(misma regla semántica)} \\[8pt]
inc\_dec\_op &\rightarrow \mathbf{++} \;\{inc\_dec\_op.op = \text{``++''}\} \\
            &\;\mid\; \mathbf{-{}-} \;\{inc\_dec\_op.op = \text{``-{}-''}\}
\end{align*}


\subsection{Análisis semántico}

El análisis semántico es la fase siguiente del sintáctico,
donde este recibe el AST generado y recorriéndolo de forma recursiva para aplicar las reglas de validación.
la parte fundamental es la tabla de simbolos y unas funciones auxiliares que permiten inferir tipos, 
verificar compatibilidad y validar las construcciones del lenguaje.

\subsubsection{Tabla de símbolos}

La tabla de símbolos se implementa como una pila de diccionarios,
donde cada diccionario representa un (\textit{scope}) de variables.
Cuando esta parte entra a un bloque de código como lo es el bloque de un \texttt{if}, \texttt{while} o \texttt{for}, 
se crea un nuevo scope apilando un diccionario vacío. 
Al salir del bloque, se elimina el scope superior, despues las operaciones principales de la tabla son:
\textit{declare}, que registra una variable verificando que no exista redeclaración, 
\textit{lookup}, que busca una variable recorriendo los scopes desde el más interno hasta el más externo, 
y \textit{snapshot}, que hace que tengamos una vista combinada de todas las variables visibles en 
el punto actual del análisis, esto para los mensajes de error del compilador.

\subsubsection{Reglas semánticas implementadas}

Las reglas semánticas que el compilador valida son: 
\begin{itemize}
    \item Primero, que toda variable sea declarada antes de ser utilizada en cualquier expresión o instrucción de entrada/salida.
    \item Segundo, la compatibilidad de tipos en las asignaciones donde al asignar un \texttt{int} a un \texttt{float} es válido, pero no al revés.
    \item Tercero, que las operaciones aritméticas requieren operandos numéricos y las operaciones lógicas requieren tipos evaluables como booleanos. 
    \item Cuarto, la validación de las instrucciones \texttt{printf} y \texttt{scanf}, donde se revusa que la cantidad de especificadores de formato sea la misma
    (\texttt{\%d} con \texttt{int}, \texttt{\%f} con \texttt{float}/\texttt{double}, \texttt{\%c} con \texttt{char}).
    \item  Quinto, que las condiciones utilizadas en las estructuras \texttt{if}, \texttt{while} y \texttt{for} sean expresiones que evaluen como valores booleanos de true o false.
\end{itemize}


\subsubsection{Funciones auxiliares de inferencia de tipos}

El parte de la semantica utiliza funciones auxiliares para la inferencia y verificación de tipos.
La función de inferencia de tipo literal determina si un valor es de tipo carácter, esto lo hace por medio de ver su inicio y final, viendo si tiene comillas simples
o cadena si está delimitado por comillas dobles. 

La función de inferencia de tipo constante determina si un número es entero o flotante según la presencia de punto decimal. 

La función de verificación numérica valida si un tipo pertenece a alguno de los tipos numéricos como lo son el
(\texttt{int}, \texttt{float}, \texttt{double}, \texttt{char}). 

La función de evaluabilidad lógica verifica si un tipo puede usarse como condición booleana, 
donde ademas acepta el tipo \texttt{bool}. 

La función de compatibilidad de asignación implementa la revision de que tipo de datos aceptan otro tipo de datos: \texttt{double} acepta \texttt{float}, \texttt{int} y \texttt{char}; \texttt{float} acepta \texttt{int} y \texttt{char}; \texttt{int} acepta \texttt{char}. 

La función de extracción de especificadores de formato revisa las cadenas de formato de 
\texttt{printf}/\texttt{scanf} para identificar los tipos de datos que se van tanto a leer como a imprimir
(\texttt{\%d}, \texttt{\%f}, \texttt{\%c}, \texttt{\%s}) ignorando los escapes.

\subsubsection{Proceso de recorrido del AST}

El análisis semántico se realiza mediante dos funciones recursivas que recorren el AST. 

La primera función determina el tipo de una expresión esto lo hace evaluando recursivamente los arboles generados a 
apartir del arbol principal, esto para identificadores, consulta la tabla de símbolos,
para constantes, infiere el tipo a partir del valor, pero para operaciones binarias, 
verifica que ambos operandos sean válidos y aplica la promoción de tipos. 

La segunda función recorre cada nodo del AST aplicando las reglas semánticas correspondientes según el tipo de nodo: 
para declaraciones, registra la variable en la tabla de símbolos y verifica la compatibilidad del inicializador, pero para asignaciones,
verifica que la variable exista y que el tipo sea compatible, despues 
para estructuras de control, revisa que la condición es correcta, ahora para la parte de instrucciones de E/S,
verifica los especificadores de formato y los argumentos.


\subsection{Entrada/Salida y ejecución}

El programa soporta dos modos de ejecución: por línea de comandos (modo terminal) y mediante una interfaz gráfica de usuario implementada con Streamlit.

En el modo terminal, el programa presenta un menú interactivo donde el usuario puede elegir entre ingresar código C directamente por teclado o cargar un archivo desde la carpeta de entradas. El programa procesa el análisis léxico, sintáctico y semántico de manera secuencial, y muestra un reporte detallado que incluye la tabla de tokens, los tokens agrupados por tipo, el resultado del análisis sintáctico, los errores encontrados (si los hay) y la traza del análisis semántico. En caso de que se detecte un error sintáctico, el análisis semántico no se ejecuta, ya que no es posible construir un AST válido. El programa permite ejecutar múltiples análisis de forma consecutiva sin reiniciar la aplicación.

En el modo de interfaz gráfica, se ofrece la misma funcionalidad pero con una presentación más visual e interactiva. Cada fase del análisis se puede visualizar de forma organizada con colores y formatos que facilitan la lectura. Adicionalmente, la UI permite visualizar el grafo del AST generado, el cual soporta zoom para examinar árboles de gran tamaño.

La cadena de formato contiene dos especificadores (\texttt{\%d \%d}), pero solo se proporciona un argumento (\texttt{x}). La regla SDT de \texttt{printf} extrae los especificadores y verifica la correspondencia. Se reporta: \texttt{Semantic error: printf expected 2 args but got 1 at line 2}. \\
\hline
Error semántico. Formato incompatible en \texttt{scanf} &
\texttt{char c = 'a';} \newline \texttt{scanf("\%d", \&c);} &
El especificador \texttt{\%d} requiere una variable de tipo \texttt{int}, pero \texttt{c} fue declarada como \texttt{char}. La función de compatibilidad de formato verifica que \texttt{\%d} solo es compatible con \texttt{int}. Se reporta: \texttt{Semantic error: scanf format '\%d' incompatible with 'char' at line 2}. \\
\hline
Error semántico. Condición con variable no declarada &
\texttt{int x = 1;} \newline \texttt{if (q > 0) \{} \newline \texttt{\quad x = x + 1;} \newline \texttt{\}} &
La variable \texttt{q} no existe en la tabla de símbolos. Al evaluar la condición del \texttt{if}, la búsqueda falla antes de poder verificar la operación relacional. Se reporta: \texttt{Semantic error: variable 'q' not declared at line 2}. \\
\hline
\end{longtable}

Los nueve casos de prueba producen exactamente el resultado esperado. En los casos con error sintáctico, el análisis semántico no se ejecuta ya que no es posible construir un AST válido. En todos los casos semánticos, la sintaxis es válida y el error es detectado correctamente por las reglas SDT durante el recorrido del AST, generando mensajes que incluyen el tipo de error, la variable o token involucrado y el número de línea correspondiente.
