\section{Introduction} %


This document contains the necessary documentation for the comprehesion of  the construction of a syntactic analyzer and a Syntax-Directed Translation (SDT) , which extend the lexical analyzer developed in the previous phase of this project. The system processes source code written in a subset of the C language, performing syntactic analysis through a bottom-up LALR(1) parser and semantic validation by means of SDT rules applied over the resulting Abstract Syntax Tree (AST).

The lexical phase, previously implemented using the Python Pygments library, produces a token flow that serves as input to the syntactic phase. Since PLY.YACC  the tool selected for the LALR(1) parser  requires a specific lexer interface, an adapter module was developed to integrate both components. The parser then constructs an AST that is later traversed by the semantic analyzer, which enforces the typing and scoping rules of the language.

We should say that , as in the previous phase, the use of the FLEX and BISON tools is not permitted. Consequently, PLY (Python Lex-Yacc) was selected as the parser generator, which imposes specific requirements on both the grammar structure and the lexer output format, leading to a significant redesign of the Context-Free Grammar defined in the previous phase.




\subsection*{Objectives}

\begin{itemize}
    \item Extend the lexical analyzer developed in the previous phase into a full syntactic and semantic analysis for a subset of the C language.
    \item Adapt the Context-Free Grammar defined in the previous phase to be compatible with PLY.YACC's LALR(1) bottom-up parsing strategy, converting right-recursive productions to left-recursive form and delegating operator precedence to PLY's precedence mechanism.
    \item Construct an Abstract Syntax Tree (AST) during syntactic analysis by associating semantic actions with each grammar production rule, following the Syntax-Directed Translation approach.
    \item Implement a symbol table with scope management to enforce semantic rules, including variable declaration before use, type compatibility in assignments, valid operands in expressions, and correct argument usage in printf and scanf.
    \item Provide both a terminal-based and a Streamlit graphical interface for code input, analysis execution, and result visualization, including an interactive AST graph view.
\end{itemize}

\section{Theorical Framework}
Firstly, the construction of a syntactic and semantic analyzer requires a solid understanding of several theoretical concepts that form the foundation of compiler design. [1]

In this project we address the intermediate stages of the front-end of the compiler: syntactic analysis and semantic validation through Syntax-Directed Translation. These phases work together to verify that the source code provided by the user is both structurally and semantically correct before proceeding with any code generation. As in the previous phase, the validation process is applied to code written in C that uses a specific set of instructions.


\subsection*{2.1 Syntactic Analysis (Parsing) and LALR(1)}

The syntactic analyzer is the second stage of the compiler. It receives the token flow generated by the lexical analyzer and verifies that the sequence in order to accomplish the structural rules of the language, as defined by a Context-Free Grammar (CFG). [1]

In the previous phase of this project, a Top-Down Recursive Descent Parser was implemented, which constructs the parse tree starting from the initial grammar symbol and expanding downward toward the tokens. This approach required right-recursive production rules and explicit operator precedence layering within the grammar .

The current phase replaces this approach with a Bottom-Up LALR(1) parser, as implemented by PLY.YACC. In bottom-up parsing, the parse tree is constructed from the leaves (tokens) upward to the root (start symbol) through a sequence of shift and reduce operations. LALR(1) (Look-Ahead LR with 1 token of lookahead) is a widely used variant, it combines similar parser states to make a smaller and more efficient parsing table, while still working for most real programming languages.

A big advantage is that it handles left-recursive rules (which cause problems in top-down parsers) without issues in fact, they help make parsing faster.

Also, PLY.YACC lets you directly define operator precedence and associativity, so you don’t have to complicate your grammar to handle them.

\subsection*{2.2 Grammar Adaptation for LALR(1)}

A Context-Free Grammar (CFG) is a way to describe how a language is built. It is defined by four parts: G=(V,T,P,S), where V is the set of non-terminal symbols (things that can be expanded), T is the set of terminal symbols (actual tokens of the language), P is the set of production rules written like A $\rightarrow$ $\alpha$, and S is the start symbol. [3]

In the previous phase, the grammar used right-recursive rules to avoid left recursion, since top-down parsers cannot handle left recursion. For the LALR(1) parser, these rules were changed to left-recursive form, as recommended by PLY. For example:

\begin{center} 

\texttt{stmt\_list $\rightarrow$ stmt | stmt stmt\_list} \quad

\textit{(right-recursive, previous phase)}\\ 

\texttt{stmt\_list $\rightarrow$ stmt\_list stmt | stmt} \quad 

\textit{(left-recursive, current phase)} 

\end{center}

Also, operator precedence (like for arithmetic and logical operations) was removed from the grammar rules and instead defined using PLY’s precedence tuple. This lets you specify priority and associativity (left, right, or none) in a simpler way, making the grammar cleaner.

Finally, epsilon (empty) productions were used as little as possible. When they were necessary—such as in optional parts of a \texttt{for}-loop—an explicit empty rule was created using a non-terminal called \texttt{empty}, since PLY.YACC does not have a special way to write epsilon directly. [2]

\subsection*{2.3 Abstract Syntax Tree (AST)}

The main goal of the parser is to take a sequence of tokens and organize them into a tree structure called an Abstract Syntax Tree (AST). Unlike a full parse tree, the AST removes unnecessary details like parentheses and punctuation, keeping only the important parts that give meaning to the program. This makes it easier to understand how the code is structured and to check that it follows the correct syntax. [1]

In this project, each node of the AST contains four things: the \textit{kind} (the type of construct, like \texttt{"assign\_stmt"} or \texttt{"if\_stmt"}), the \textit{children} (a list of child nodes or values), the \textit{lineno} (the line number in the source code, useful for errors), and the \textit{value} (the actual text for terminal symbols). The AST is built step by step during parsing, where each grammar rule creates its corresponding node using a function called \texttt{make\_node}.[6]

\subsection*{2.4 Syntax-Directed Translation (SDT)}

Syntax-Directed Translation (SDT) is a method where grammar rules are linked with actions that give them meaning. For each rule like A $\rightarrow$ $\alpha$, some actions are executed when the parser uses that rule. These actions usually compute values (called synthesized attributes) that move from the child nodes up to the parent node. [1]

In this project, SDT is used in two steps. First, during parsing, each grammar rule creates an AST node as its result. Then, after parsing, a separate semantic analyzer goes through the AST and checks meanings by applying rules to each node. This two-step process separates building the structure from checking its correctness.

During semantic analysis, a detailed SDT trace is kept. For each node visited, it records the step number, the type of node, the line number, the action performed, and the state of the symbol table at that moment. This makes it easier to see what the program is doing and helps with debugging.[5]

\subsection*{2.5 Symbol Table and Scope Management}

A symbol table is a data structure used during compilation to store information about variables (identifiers) in the program. At least, it keeps the name of each variable and its type. It is used during semantic analysis to check that variables are declared before being used and that their types are used correctly. [1]

Scope management is handled using a stack of dictionaries, where each dictionary represents a different block of code. When the parser enters a new block (like inside an \texttt{if}, \texttt{while}, or \texttt{for}), a new scope is added to the stack. When leaving the block, that scope is removed. This makes sure that variables only exist inside the block where they were declared. When searching for a variable, the system looks from the most recent scope to the oldest one, following the usual scoping rules of C.[8]

\subsection*{2.6 Semantic Validation Rules}

Semantic analysis checks that a program makes sense, even if it is written with correct syntax. In this project, the following simple rules are applied:

\begin{itemize}
\item \textbf{Declaration before use:} A variable must be declared before it is used. If not, an “undeclared variable” error is reported with the line number.

\item \textbf{Type compatibility in assignment:} The value assigned to a variable must match its type. Types follow this order for compatibility: \texttt{char} $\le$ \texttt{int} $\le$ \texttt{float} $\le$ \texttt{double}.

\item \textbf{Valid operands in expressions:} Values used in expressions must be valid types like numbers or characters. If a variable is not declared, it causes an error that affects the whole expression.

\item \textbf{\texttt{printf} and \texttt{scanf} argument validation:} The number and types of arguments must match the format specifiers. For example, \texttt{\%d} is for \texttt{int}, \texttt{\%f} for \texttt{float}/\texttt{double}, \texttt{\%c} for \texttt{char}, and \texttt{\%s} for strings. If they don’t match, an error is generated.

\item \textbf{Evaluable conditions:} Conditions in \texttt{if}, \texttt{while}, and \texttt{for} must be of a type that can be evaluated as true or false, such as \texttt{bool}, \texttt{int}, \texttt{float}, \texttt{double}, or \texttt{char}.

\end{itemize}

\subsection*{2.7 Lexer Adapter and PLY Compatibility}

PLY.YACC expects the lexer to have a \texttt{token()} method that returns one token at a time, where each token has \texttt{.type} and \texttt{.value}. When there are no more tokens, it should return \texttt{None}. However, the lexer used in this project (based on Pygments) gives a list of \texttt{(type, value)} pairs, which is not directly compatible.[7]

To fix this, an Adapter pattern was used. The class \texttt{PlyLexerAdapter} takes the list of tokens, converts the type names to the uppercase format expected by PLY, and provides a \texttt{token()} method that returns tokens one by one.

This way, the difference between the lexer and the parser is handled in one place, so both parts can work correctly without needing to change their original design. [4]


\section{Development}

\subsection{Implementation}

The compiler folowed an architecture divided into three layers, where first we have the logic layer, then we have the presentation layer wich is the frontend and finaly we have the utilities layer, for this the logic layer is composed of six sections, the first one is the lexical analyzer, the token adapter, the grammar rules, the AST structures, the semantic validation rules and the main phase that integrates the complete logic of syntactic and semantic analysis.

The flow folows a pipeline in a sequential way, where first, the lexical analyzer tokenizes the C source code using Pygments as a base. After this, a secondary block transforms the lexer output to the format require by PLY (Python Lex-Yacc) and then, the PLY LALR(1) parser analizes the tokens and builds the AST. Finaly, the semantic module traverses the AST aplying the defined validation rules.

The built parser uses PLY to generate a LALR(1) analyzer from these productions.

\subsubsection{Justification of the productions}

The grammer is composed of 25 non-terminals and 60 productions that define the language:
\vspace{0.5cm}

\textbf{Program structure:} Rule 1 defines that a program consists of a list of statements, wich is built recursivley from the left using Rules 2 and 3.

\textbf{Statements:} Rules 4 through 12 establish that each statment can be one of nine alternatives: numeric declaration, character declaration, assignment, \texttt{if} conditional, \texttt{while} loop, \texttt{for} loop, printing with \texttt{printf}, reading with \texttt{scanf}.

\textbf{Declarations:} Rules 13 through 14 define numeric variable declarations (\texttt{int}, \texttt{float}, \texttt{double}), both simple and with initialization. Rules 15 and 16 do the same but for the character type (Rule 18), where initialization only accepts character literals.

\textbf{Assignments:} Rule 19 defines assignment with simple or compound operators (Rule 21: \texttt{=}, \texttt{+=}, \texttt{-=}, \texttt{/=}) over the aritmethic part and Rule 20 allows direct assignment of characters.

\textbf{Control structures:} Rules 22 and 23 define control blocks, such as \texttt{if} and \texttt{while}, wich require a condition in parentheses and a block that starts and ends with curly braces. Rule 24 defines the \texttt{for} loop with three parts seperated by semicolons: initialization (Rules 25--28), condition (Rules 29--30) and step (Rules 31--34), each of wich can be empty ($\varepsilon$).

\textbf{Input/Output:} Rule 35 indicates the \texttt{printf} instruction, whose arguments are specifyed in Rules 36--37. Rule 38 indicates \texttt{scanf}, wich accepts a format string and an identifier that must be followed by \texttt{\&}. 

\textbf{Arithmetic expressions:} Rules 44 through 51 indicate mathematical expresions with five operators, identifiers, numeric constants and grouping with parentheses. 

\textbf{Conditions:} Rules 52 through 58 define conditions with logical operators, negation, relational comparisons (Rule 59), grouping with parentheses and the conversion of arithmetic expresions into booleans.

\subsubsection{Parse tree}

To give an example of how the LALR(1) works, the parse tree generated for the following input is used:

\begin{lstlisting}
x = x + 1;
\end{lstlisting}

The LALR(1) parser implemented with PLY performs shift operations of tokens onto the stack and reduces when it recognizes the right side of a production at the top of the stack.

The bottom-up process for this input works as folows, where first the analyzer shifts the token \texttt{x} onto the stack, then shifts the token \texttt{=} and reduces it to the non-terminal using Rule 21. Then it shifts the second token \texttt{x} and reduces it to \textit{math\_expr} thru Rule 49. Then it shifts \texttt{+} and the token \texttt{1}, wich is reduced to \textit{math\_expr} by Rule 50.

At this point, the parser has at the top of the stack the following elements:

$$math\_expr \; \texttt{+} \; math\_expr$$ 

Wich is the same as the right-hand side of Rule 44.

\newpage

With this said, this reduction is applied wich generates a \textit{math\_expr} node that represents the expresion $x + 1$. Finaly the token \texttt{;} is shifted and the stack contains the elements 

$$\texttt{IDENTIFIER} \; assign\_expr\_op \; math\_expr \; \texttt{;}$$ 

And this is the same as with Rule 19 and this reduction is applied creating the \textit{assign\_stmt} node, wich is reduced to \textit{stmt} by Rule 6, then to \textit{stmt\_list} by Rule 3, and finaly to the initial symbol \textit{program} thru Rule 1.

\subsection{Syntax-Directed Translation (SDT)}

Syntax-Directed Translation (SDT) links semantic rules to each production of the grammarto each production of the grammer. 

For this, with the working approach already established, we can implement and present the SDT rules for each production, where:

$SymTable$ represents the symbol table. 

\textit{.type} is the infered type attribute.

\textit{.node} is the generated AST node.

\subsubsection{SDT Rules -- Program structure}

\begin{align*}
program &\rightarrow stmt\_list \quad \{program.node = \text{make\_node}(\text{``program''}, stmt\_list.nodes)\} \\[8pt]
stmt\_list &\rightarrow stmt\_list_1 \; stmt \quad \{stmt\_list.nodes = stmt\_list_1.nodes \cup \{stmt.node\}\} \\[4pt]
stmt\_list &\rightarrow stmt \quad \{stmt\_list.nodes = \{stmt.node\}\} \\[8pt]
stmt &\rightarrow decl\_num \mid decl\_char \mid assign\_stmt \mid if\_stmt \\
     &\quad\mid while\_stmt \mid for\_stmt \mid print\_stmt \mid scan\_stmt \\
     &\quad\mid unary\_stmt \quad \{stmt.node = X.node\}
\end{align*}

\subsubsection{SDT Rules -- Declarations}

\begin{align*}
decl\_num &\rightarrow type\_num \; \mathbf{id} \; \mathbf{;} \\
         &\quad \{\textbf{if } SymTable.\text{declare}(\mathbf{id}.name, type\_num.type) \textbf{ then } ok \\
         &\quad \textbf{ else } \text{error}(\text{``variable redeclaration''})\} \\[10pt]
decl\_num &\rightarrow type\_num \; \mathbf{id} \; \mathbf{=} \; math\_expr \; \mathbf{;} \\
         &\quad \{\textbf{if } SymTable.\text{declare}(\mathbf{id}.name, type\_num.type) \\
         &\quad \textbf{ then if } \text{compatible}(type\_num.type, math\_expr.type) \textbf{ then } ok \\
         &\quad \textbf{ else } \text{error}(\text{``incompatible type in assignment''}) \\
         &\quad \textbf{ else } \text{error}(\text{``variable redeclaration''})\} \\[10pt]
decl\_char &\rightarrow type\_char \; \mathbf{id} \; \mathbf{;} \\
          &\quad \{\textbf{if } SymTable.\text{declare}(\mathbf{id}.name, \text{``char''}) \textbf{ then } ok \\
          &\quad \textbf{ else } \text{error}(\text{``redeclaration''})\} \\[10pt]
decl\_char &\rightarrow type\_char \; \mathbf{id} \; \mathbf{=} \; \mathbf{VALUE\_CHAR} \; \mathbf{;} \\
          &\quad \{\textbf{if } SymTable.\text{declare}(\mathbf{id}.name, \text{``char''}) \textbf{ then } ok \\
          &\quad \textbf{ else } \text{error}(\text{``redeclaration''})\} \\[10pt]
type\_num &\rightarrow \mathbf{int} \quad \{type\_num.type = \text{``int''}\} \\
type\_num &\rightarrow \mathbf{float} \quad \{type\_num.type = \text{``float''}\} \\
type\_num &\rightarrow \mathbf{double} \quad \{type\_num.type = \text{``double''}\} \\[6pt]
type\_char &\rightarrow \mathbf{char} \quad \{type\_char.type = \text{``char''}\}
\end{align*}

\subsubsection{SDT Rules -- Assignment}

\begin{align*}
assign\_stmt &\rightarrow \mathbf{id} \; assign\_op \; math\_expr \; \mathbf{;} \\
            &\quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{null} \\
            &\quad \textbf{ then } \text{error}(\text{``variable not declared''}) \\
            &\quad \textbf{ else if } assign\_op.op = \text{``=''} \\
            &\quad\quad \textbf{ then if } \text{compatible}(SymTable.\text{lookup}(\mathbf{id}.name), math\_expr.type) \\
            &\quad\quad \textbf{ then } ok \textbf{ else } \text{error}(\text{``incompatible type''}) \\
            &\quad \textbf{ else if } assign\_op.op \in \{\text{``+=''}, \text{``-=''}, \text{``/=''}\} \\
            &\quad\quad \textbf{ then if } \text{numeric}(SymTable.\text{lookup}(\mathbf{id}.name)) \land \text{numeric}(math\_expr.type) \\
            &\quad\quad \textbf{ then } ok \textbf{ else } \text{error}(\text{``non-numeric operands''})\} \\[10pt]
assign\_stmt &\rightarrow \mathbf{id} \; \mathbf{=} \; \mathbf{VALUE\_CHAR} \; \mathbf{;} \\
            &\quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{null} \\
            &\quad \textbf{ then } \text{error}(\text{``variable not declared''}) \\
            &\quad \textbf{ else if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{``char''} \textbf{ then } ok \\
            &\quad \textbf{ else } \text{error}(\text{``incompatible type''})\} \\[10pt]
assign\_op &\rightarrow \mathbf{=} \;\{assign\_op.op = \text{``=''}\} \\
           &\mid \mathbf{+=} \;\{assign\_op.op = \text{``+=''}\} \\
           &\mid \mathbf{-=} \;\{assign\_op.op = \text{``-=''}\} \\
           &\mid \mathbf{/=} \;\{assign\_op.op = \text{``/=''}\}
\end{align*}

\subsubsection{SDT Rules -- Arithmetic expressions}

\begin{align*}
math\_expr &\rightarrow math\_expr_1 \; \mathbf{+} \; math\_expr_2 \\
           &\quad \{\textbf{if } \text{numeric}(math\_expr_1.type) \land \text{numeric}(math\_expr_2.type) \\
           &\quad \textbf{ then } math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \\
           &\quad \textbf{ else } \text{error}(\text{``non-numeric operands''})\} \\[6pt]
math\_expr &\rightarrow math\_expr_1 \; \mathbf{-} \; math\_expr_2 \\
           &\quad \{math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \text{ (same rule)}\} \\[6pt]
math\_expr &\rightarrow math\_expr_1 \; \mathbf{*} \; math\_expr_2 \\
           &\quad \{math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \text{ (same rule)}\} \\[6pt]
math\_expr &\rightarrow math\_expr_1 \; \mathbf{/} \; math\_expr_2 \\
           &\quad \{math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \text{ (same rule)}\} \\[6pt]
math\_expr &\rightarrow math\_expr_1 \; \mathbf{\%} \; math\_expr_2 \\
           &\quad \{math\_expr.type = \text{promote}(math\_expr_1.type, math\_expr_2.type) \text{ (same rule)}\} \\[10pt]
math\_expr &\rightarrow \mathbf{id} \quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) \neq \text{null} \\
           &\quad \textbf{ then } math\_expr.type = SymTable.\text{lookup}(\mathbf{id}.name) \\
           &\quad \textbf{ else } \text{error}(\text{``variable not declared''})\} \\[6pt]
math\_expr &\rightarrow \mathbf{VALUE\_NUM} \quad \{math\_expr.type = \text{infer}(\mathbf{VALUE\_NUM}.value)\} \\[6pt]
math\_expr &\rightarrow \mathbf{(} \; math\_expr_1 \; \mathbf{)} \quad \{math\_expr.type = math\_expr_1.type\}
\end{align*}

\subsubsection{SDT Rules -- Conditions}

\begin{align*}
cond &\rightarrow cond_1 \; \mathbf{||} \; cond_2 \\
     &\quad \{\textbf{if } \text{logical}(cond_1.type) \land \text{logical}(cond_2.type) \\
     &\quad \textbf{ then } cond.type = \text{``bool''} \\
     &\quad \textbf{ else } \text{error}(\text{``invalid operands for logical OR''})\} \\[6pt]
cond &\rightarrow cond_1 \; \mathbf{\&\&} \; cond_2 \\
     &\quad \{\textbf{if } \text{logical}(cond_1.type) \land \text{logical}(cond_2.type) \\
     &\quad \textbf{ then } cond.type = \text{``bool''} \\
     &\quad \textbf{ else } \text{error}(\text{``invalid operands for logical AND''})\} \\[6pt]
cond &\rightarrow math\_expr_1 \; rel\_op \; math\_expr_2 \\
     &\quad \{\textbf{if } \text{numeric}(math\_expr_1.type) \land \text{numeric}(math\_expr_2.type) \\
     &\quad \textbf{ then } cond.type = \text{``bool''} \\
     &\quad \textbf{ else } \text{error}(\text{``non-numeric operands in comparison''})\} \\[6pt]
cond &\rightarrow \mathbf{!} \; cond_1 \\
     &\quad \{\textbf{if } \text{logical}(cond_1.type) \textbf{ then } cond.type = \text{``bool''} \\
     &\quad \textbf{ else } \text{error}(\text{``invalid operand for negation''})\} \\[6pt]
cond &\rightarrow \mathbf{(} \; cond_1 \; \mathbf{)} \quad \{cond.type = cond_1.type\} \\[6pt]
cond &\rightarrow math\_expr \quad \{cond.type = math\_expr.type\} \\[6pt]
cond &\rightarrow \mathbf{VALUE\_CHAR} \quad \{cond.type = \text{``char''}\}
\end{align*}

Where the $\text{logical}(t)$ function returns true if $t \in \{\text{``bool'', ``int'', ``float'', ``double'', ``char''}\}$, these are the data types that are evaluated as a logical function.

\begin{align*}
rel\_op &\rightarrow \mathbf{==} \;\{rel\_op.op = \text{``==''}\} \;\mid\; \mathbf{!=} \;\{rel\_op.op = \text{``!=''}\} \\
        &\;\mid\; \mathbf{<} \;\{rel\_op.op = \text{``<''}\} \;\mid\; \mathbf{>} \;\{rel\_op.op = \text{``>''}\} \\
        &\;\mid\; \mathbf{<=} \;\{rel\_op.op = \text{``<=''}\} \;\mid\; \mathbf{>=} \;\{rel\_op.op = \text{``>=''}\}
\end{align*}

\subsubsection{SDT Rules -- Control structures}

\begin{align*}
if\_stmt &\rightarrow \mathbf{if} \; \mathbf{(} \; cond \; \mathbf{)} \; \mathbf{\{} \; stmt\_list \; \mathbf{\}} \\
        &\quad \{\textbf{if } \text{logical}(cond.type) \textbf{ then } ok \\
        &\quad \textbf{ else } \text{error}(\text{``condition not evaluable in if''})\} \\[10pt]
while\_stmt &\rightarrow \mathbf{while} \; \mathbf{(} \; cond \; \mathbf{)} \; \mathbf{\{} \; stmt\_list \; \mathbf{\}} \\
           &\quad \{\textbf{if } \text{logical}(cond.type) \textbf{ then } ok \\
           &\quad \textbf{ else } \text{error}(\text{``condition not evaluable in while''})\} \\[10pt]
for\_stmt &\rightarrow \mathbf{for} \; \mathbf{(} \; for\_init \; \mathbf{;} \; for\_cond \; \mathbf{;} \; for\_step \; \mathbf{)} \; \mathbf{\{} \; stmt\_list \; \mathbf{\}} \\
         &\quad \{SymTable.\text{enter\_scope}(); \\
         &\quad \text{evaluate}(for\_init); \\
         &\quad \textbf{if } for\_cond \neq \varepsilon \land \neg\text{logical}(for\_cond.type) \\
         &\quad \textbf{ then } \text{error}(\text{``condition not evaluable in for''}); \\
         &\quad \text{evaluate}(for\_step); \; \text{evaluate}(stmt\_list); \\
         &\quad SymTable.\text{exit\_scope}()\}
\end{align*}

\subsubsection{SDT Rules -- Input/Output}

\begin{align*}
print\_stmt &\rightarrow \mathbf{printf} \; \mathbf{(} \; print\_args \; \mathbf{)} \; \mathbf{;} \\
           &\quad \{\text{let } specs = \text{extract\_specs}(\mathbf{TEXT\_STRING}.value) \\
           &\quad \textbf{if } |specs| \neq |id\_list| \textbf{ then } \text{error}(\text{``incorrect number of arguments''}) \\
           &\quad \textbf{else } \forall (spec_i, id_i) \in \text{zip}(specs, id\_list): \\
           &\quad\quad \textbf{if } SymTable.\text{lookup}(id_i) = \text{null} \\
           &\quad\quad \textbf{ then } \text{error}(\text{``variable not declared''}) \\
           &\quad\quad \textbf{ else if } \neg\text{format\_ok}(SymTable.\text{lookup}(id_i), spec_i) \\
           &\quad\quad \textbf{ then } \text{error}(\text{``incompatible format''})\} \\[10pt]
scan\_stmt &\rightarrow \mathbf{scanf} \; \mathbf{(} \; \mathbf{TEXT\_STRING} \; \mathbf{,} \; \mathbf{\&} \; \mathbf{id} \; \mathbf{)} \; \mathbf{;} \\
          &\quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{null} \\
          &\quad \textbf{ then } \text{error}(\text{``variable not declared''}) \\
          &\quad \textbf{ else } \text{let } specs = \text{extract\_specs}(\mathbf{TEXT\_STRING}.value) \\
          &\quad \textbf{if } |specs| \neq 1 \textbf{ then } \text{error}(\text{``exactly one specifier is expected''}) \\
          &\quad \textbf{ else if } \neg\text{format\_ok}(SymTable.\text{lookup}(\mathbf{id}.name), specs[0]) \\
          &\quad \textbf{ then } \text{error}(\text{``format incompatible with variable type''})\}
\end{align*}

Where the $\text{format\_ok}$ function verifies that a variable type and a format specifier are compatible: \texttt{\%d}/\texttt{\%i} needs an \texttt{int} variable, \texttt{\%f} needs a \texttt{float} variable, to put them as an example.

\subsubsection{SDT Rules -- Unary statements}

\begin{align*}
unary\_stmt &\rightarrow inc\_dec\_op \; \mathbf{id} \; \mathbf{;} \\
           &\quad \{\textbf{if } SymTable.\text{lookup}(\mathbf{id}.name) = \text{null} \\
           &\quad \textbf{ then } \text{error}(\text{``variable not declared''}) \\
           &\quad \textbf{ else if } \neg\text{numeric}(SymTable.\text{lookup}(\mathbf{id}.name)) \\
           &\quad \textbf{ then } \text{error}(\text{``unary operator requires numeric type''})\} \\[6pt]
unary\_stmt &\rightarrow \mathbf{id} \; inc\_dec\_op \; \mathbf{;} \quad \text{(same semantic rule)} \\[8pt]
inc\_dec\_op &\rightarrow \mathbf{++} \;\{inc\_dec\_op.op = \text{``++''}\} \\
            &\;\mid\; \mathbf{-{}-} \;\{inc\_dec\_op.op = \text{``-{}-''}\}
\end{align*}


\subsection{Semantic analysis}

The semantic analysis is the phase that comes after the syntactic one, where it receives the generated AST and traverses it recursivley to apply the validation rules. the fundamental part is the symbol table and some auxiliary functions that allow infering types, verifing compatibility and validating the language constructs.

\subsubsection{Symbol table}

The symbol table is implemented as a stack of dictionaries, where each dictionary represents a (\textit{scope}) of variables. When this part enters a code block such as the block of an \texttt{if}, \texttt{while} or \texttt{for}, a new scope is created by pushing an empty dictionary. When leaving the block, the top scope is removed, then the main table operations are: \textit{declare}, wich registers a variable verifing that no redeclaration exists, \textit{lookup}, wich searches for a variable traversing the scopes from the innermost to the outermost, and \textit{snapshot}, wich gives us a combined view of all variables visible at the curent point of analysis, this for the compiler's error mesages.

\subsubsection{Implemented semantic rules}

The semantic rules that the compiler validates are: 
\begin{itemize}
    \item First, that every variable be declared before being used in any expression or input/output instruction.
    \item Second, type compatibility in assignments where asigning an \texttt{int} to a \texttt{float} is valid, but not the other way around.
    \item Third, that arithmetic operations require numeric operands and logical operations require types evaluable as booleans. 
    \item Fourth, the validation of \texttt{printf} and \texttt{scanf} instructions, where it is cheked that the number of format specifiers matches (\texttt{\%d} with \texttt{int}, \texttt{\%f} with \texttt{float}/\texttt{double}, \texttt{\%c} with \texttt{char}).
    \item Fifth, that the conditions used in \texttt{if}, \texttt{while} and \texttt{for} structures are expresions that evaluate as boolean values of true or false.
\end{itemize}


\subsubsection{Type inference auxiliary functions}

The semantic part uses auxiliary functions for type inference and verification. The literal type inference function determines if a value is of character type, it does this by looking at its beginning and end, seeing if it has single quotes or string if it is delimited by double quotes. 

The constant type inference function determines if a number is integer or float acording to the presence of a decimal point. 

The numeric verification function validates if a type belongs to any of the numeric types such as (\texttt{int}, \texttt{float}, \texttt{double}, \texttt{char}). 

The logical evaluability function verifies if a type can be used as a boolean condition, where it also accepts the \texttt{bool} type. 

The assignment compatibility function implements the revision of wich data types accept another data type: \texttt{double} accepts \texttt{float}, \texttt{int} and \texttt{char}; \texttt{float} accepts \texttt{int} and \texttt{char}; \texttt{int} accepts \texttt{char}. 

The format specifier extraction function reviews the format strings of \texttt{printf}/\texttt{scanf} to identify the data types that are going to be both read and printed (\texttt{\%d}, \texttt{\%f}, \texttt{\%c}, \texttt{\%s}) ignoring the escape sequences.

\subsubsection{AST traversal process}

The semantic analysis is performed thru two recursive functions that traverse the AST. 

The first function determines the type of an expresion by recursivley evaluating the trees generated from the main tree, for identifiers, it queries the symbol table, for constants, it infers the type from the value, but for binary operations, it verifies that both operands are valid and aplies type promotion. 

The second function traverses each node of the AST aplying the coresponding semantic rules acording to the node type: for declarations, it registers the variable in the symbol table and verifies the compatibility of the initializer, but for assignments, it verifies that the variable exists and that the type is compatible, then for control structures, it checks that the condition is correct, now for the I/O instruction part, it verifies the format specifiers and the arguments.


\subsection{Input/Output and execution}

The program supports two execution modes: by command line (terminal mode) and through a graphical user interface implemented with Streamlit.

In terminal mode, the program presents an interactive menu where the user can choose between entering C code directly from the keyboard or loading a file from the input folder. The program processes the lexical, syntactic and semantic analysis in a sequential way, and shows a detailed report that includes the token table, tokens grouped by type, the result of the syntactic analysis, the errors found (if any) and the semantic analysis trace. In case a syntactic error is detected, the semantic analysis is not executed, since it is not posible to build a valid AST. The program allows running multiple analyses consecutivley without restarting the application.

In graphical interface mode, the same functionality is offered but with a more visual and interactive presentation. Each phase of the analysis can be visualized in an organized way with colors and formats that facilitate reading. Aditionally, the UI allows visualizing the graph of the generated AST, wich supports zoom to examine large trees.
