\section{Introduction} %

This document contains the necessary documentation for the understanding in the construction of a lexical analyzer, where it will process source code by reading it from a file or text string. This program will be able to identify, classify and count the tokens in the code that was provided to it, in addition these will be separated into categories such as keywords, identifiers, operators, punctuators, constants and literals.

However, the lexer program meets the restriction of not making use of the FLEX tool, so a lexical analyzer was implemented using the Python Pygments library, which provides a lexer for the C language.

Now then, the lexical analyzer represents the initial phase of a compilation process, where this sets the foundations for the formulation and improvement of a context-free grammar (CFG), this will use a left factorization, which later will be used for the construction of a parser, which will be responsible for validating the syntactic structure of the code that we want to compile, ensuring that it complies with the rules defined in the CFG.

\subsection*{Objectives}

\begin{itemize}
    \item Build a lexical analyzer capable of processing code through the passing of a file or a text string.
    
    \item Generate a table with the lexical analysis of the input, classifying the tokens (such as keywords, identifiers, operators, punctuators, constants and literals) and giving a count for each category.
    
    \item Provide a basic proposal of Context-Free Grammar (CFG) that represents one of the reading examples, applying left factorization to improve that grammar.
    
    \item Package the application following software engineering practices, name it as \\
\texttt{unam.fi.compilers.g5.00} and share the repository link with public permissions.

\end{itemize}

\section{Theorical Framework}
Firstly, the traduction of a high level programming languaje into a low level one is a process structured in multiple phases.

In this proyect we will aboard the inicial stages of the front-end of the compiler: the lexical analysis and the syntatic analysis. 
This phases work together in order to validate that the source code given by the user is well formed before proceding with the semantic evaluation or the code generation. In this case, the validation procees its aplied to a code that is written in C that uyses a specific set of instructions

\subsection{Lexical Analysis (Scanner / Lexer)}
The lexical analysis represents the first stage of the languaje processing. Its fundamental responsability is to redd the code character by character and group them in secuences with logical meaning,
known as lexemes.

Starting from this lexemes, the analyzer, generatees a secuence of tokens (as identifiers, key words, constants, special characters, operators, punctuation, literals).
In the proyect, the lexical analyzer identifies specific sets as:
\begin{itemize}
    \item \textbf{ID:} Identifiers used to give name to the variables
    \item \textbf{VALUE\_NUM: } Literals that represent any number whole and decimal
    \item \textbf{VALUE\_CHAR: } Literals that represent a unique character enclosed between simple commas
    \item \textbf{TEXT\_STRING: } Text strings enclosed in double quotes, mostly used as format specifiers
\end{itemize}

\subsection{Syntactic analysis (Parsing) and CFG (CONTEXT-FREE GRAMMARS)}

 The syntatic analyzer is the second stage in this compiler. It recibes the flow of tokens generate by the lexical analizer and verifies that the secuence fullfill the structural rules of the language.

 To define this rules in a rigurous way, it uses the CFG (CONTEXT-FREE GRAMMARS), it helps to stablish which rules are valid using the production rules.
 
 In this proyect, the CFG defines the structure to prodecue any instruction (StmtList), covering numeric declarations (DeclNum), assignments (AssignSmt), and flow control statements (lfStmt, WhileStmt, ForStmtm)

 An example given is the formal rule of declaring a number (and optionally initialize it) it defines matematically in the next way:


 \vspace{0.5 cm }
 \begin{center}
     $DeclNum -> TypeNum ID AssignOpNum $
 \end{center}

 \vspace{0.5 cm }

The main objective of the parser is to organize the sequence of tokens into a hierarchical structure called a parse tree (or syntax tree). This structure shows how the tokens relate to each other according to the language rules and confirms that the code follows the correct syntax

\subsection{Parsing Strategies: Top-Down Approach and Precedence Resolution}

 There are two main ways to construct the syntax tree: Top-Down and Bottom-Up. This project implements a Top-Down approach through a Recursive Descent Parser. In this method, the tree is built starting from the initial symbol of the grammar down to the leaves (the tokens), associating a recursive function with each grammar rule to process the input from left to right.

To ensure that this approach works mathematically without entering infinite loops and respecting the operation hierarchies, the grammar was designed with specific features: 

\begin{itemize}
    \item \textbf{Elimination of left recursion (Left-Recursion): } The production rules for mathematical expressions (MathExpr) were rewritten to be right-recursive.
    \item \textbf{Layered Precedence:} To evaluate logical conditions (Cond) while respecting the standardized hierarchy, the rules were divided into levels. The analysis starts at the lowest priority level looking for the OR logical operator (CondOrPrime), descends to the intermediate AND level (CondAndPrime), evaluates the level of relational comparisons (CondRel), and culminates at the highest priority level processing the logical NOT and parentheses (CondUnary). 
\end{itemize}

\subsection{Gramatica}

\subsubsection*{1. Reglas de producción }
\begin{center}
\begin{align*}
    \text{Stmt} &\rightarrow \text{Stmt} \ \text{StmtList} \mid \varepsilon \\
    \text{StmtList} &\rightarrow \text{DeclNum} \mid \text{DeclChar} \mid \text{AssignStmt} \mid \text{IfStmt} \mid \text{WhileStmt} \\
    &\mid \text{ForStmt} \mid \text{PrintStmt} \mid \text{ScanStmt} \mid \text{UnaryStmt}
\end{align*}
\end{center}

Donde:

\begin{itemize}
    \item \textbf{Stmt} es el Símbolo Inicial para generar instrucciones.
    \item \textbf{StmtList} representa las instrucciones específicas que se pueden producir.
    \item \textbf{DeclNum} y \textbf{DeclChar} son las declaraciones de números y caracteres.
    \item \textbf{AssignStmt} es para la asignación de variables.
    \item \textbf{IfStmt, WhileStmt, ForStmt} son las estructuras de control de flujo.
    \item \textbf{PrintStmt, ScanStmt} son las funciones de entrada y salida de datos.
    \item \textbf{UnaryStmt} es para las sentencias unarias (pre/post incrementos).
\end{itemize}

\subsubsection*{2. Reglas de produccion: }

\begin{align*}
    \text{DeclNum} &\rightarrow \text{TypeNum} \ \text{ID} \ \text{AssignOpNum} \text{;} \\
    \text{AssignOpNum} &\rightarrow \text{MathExpr} \mid \varepsilon \\
    \text{DeclChar} &\rightarrow \text{TypeChar} \ \text{ID} \ \text{AssignOpChar} \text{;} \\
    \text{AssignOpChar} &\rightarrow \text{VALUE\_CHAR} \mid \varepsilon \\
    \text{AssignStmt} &\rightarrow \text{ID} \ \text{AssignExprOp} \ \text{MathExpr} \mid \text{ID} \ \text{VALUE\_CHAR} \text{;} \\
    \text{TypeNum} &\rightarrow \text{int} \mid \text{float} \mid \text{double} \\
    \text{TypeChar} &\rightarrow \text{char}
\end{align*}

\newpage

Donde:

\begin{itemize}
    \item DeclNum : Es la PR para declarar un número o declarar un número e inicializarlo.
    \item DeclChar : PR para declarar un carácter o carácter un número e inicializarlo.
    \item AssignStmt : Es la PR para asignar el valor a un dato ya declarado anteriormente.
    \item TypeNum : Los tipos de datos numérico que acepta.
    \item TypeChar : Tipo de dato de carácter que acepta.
\end{itemize}


\textbf{3. Aplicando factorización por la izquierda: }

La declaración de variables $DeclNum \rightarrow TypeNum \ ID \ ; \mid TypeNum \ ID \ = \ MathExpr \ ;$ tiene el factor común $TypeNum \ ID$, por lo que podemos expresarla como:

\begin{align*}
    \text{DeclNum} &\rightarrow \text{TypeNum} \ \text{ID} \ \text{AssignOpNum} \text{;} \\
    \text{AssignOpNum} &\rightarrow \text{MathExpr} \mid \varepsilon
\end{align*}

También la expresión para operaciones continuas 
\begin{small}
\[
\text{MathExpr} \rightarrow \text{MathExpr} \ \text{MathOp} \ \text{MathOperand} \mid \text{MathOperand}
\]
\end{small}
presentaba recursividad por la izquierda, por lo que fue factorizada (recursividad por la derecha) de la siguiente manera:

\begin{small}
\begin{align*}
    \text{MathExpr} &\rightarrow \text{MathOperand} \\
    &\mid \text{MathOperand} \ \text{MathOp} \ \text{MathExpr}
\end{align*}
\end{small}

Este procedimiento ayudará a evitar el no determinismo, también con la ayuda de la recursividad por la derecha ya realizada, donde tenemos elementos no terminales en el lado derecho, por lo que esto puede ayudar con las cadenas repetitivas y recursivas que podemos hacer al anidar operaciones o llamadas lógicas.

Finalmente la gramática está definida por las expresiones:
\begin{small}
\begin{align*}
    \text{Stmt} &\rightarrow \text{Stmt} \ \text{StmtList} \mid \varepsilon \\
    \text{StmtList} &\rightarrow \text{DeclNum} \mid \text{DeclChar} \mid \text{AssignStmt} \mid \text{IfStmt} \mid \text{WhileStmt} \mid \\
    &\quad \text{ForStmt} \mid \text{PrintStmt} \mid \text{ScanStmt} \mid \text{UnaryStmt} \\
    \text{DeclNum} &\rightarrow \text{TypeNum} \ \text{ID} \ \text{AssignOpNum} \text{;} \\
    \text{AssignOpNum} &\rightarrow \text{MathExpr} \mid \varepsilon \\
    \text{DeclChar} &\rightarrow \text{TypeChar} \ \text{ID} \ \text{AssignOpChar} \text{;} \\
    \text{AssignOpChar} &\rightarrow \text{VALUE\_CHAR} \mid \varepsilon \\
    \text{AssignStmt} &\rightarrow \text{ID} \ \text{AssignExprOp} \ \text{MathExpr} \mid \text{ID} \ \text{VALUE\_CHAR} \text{;} \\
    \text{TypeNum} &\rightarrow \text{int} \mid \text{float} \mid \text{double} \\
    \text{TypeChar} &\rightarrow \text{char} \\
    \text{IfStmt} &\rightarrow \text{if} \ (\text{Cond}) \ \{ \text{StmtList} \} \\
    \text{WhileStmt} &\rightarrow \text{while} \ (\text{Cond}) \ \{ \text{StmtList} \} \\
    \text{ForStmt} &\rightarrow \text{for} \ (\text{ForInit} \text{;} \ \text{ForCond} \text{;} \ \text{ForStep}) \ \{ \text{StmtList} \} \\
    \text{ForInit} &\rightarrow \text{TypeNum} \ \text{ID} \ \text{AssignOpNum} \mid \text{ID} \ \text{AssignExprOp} \ \text{MathExpr} \mid \varepsilon \\
    \text{ForCond} &\rightarrow \text{Cond} \mid \varepsilon \\
    \text{ForStep} &\rightarrow \text{IncDecOp} \ \text{ID} \mid \text{ID} \ \text{IncDecOp} \mid \\
    &\quad \text{ID} \ \text{AssignExprOp} \ \text{MathExpr} \mid \varepsilon \\
    \text{PrintStmt} &\rightarrow \text{printf} ( \text{PrintArgs} ) \text{;} \\
    \text{PrintArgs} &\rightarrow \text{TEXT\_STRING} \mid \text{TEXT\_STRING} , \text{IdList} \text{;} \\
    \text{ScanStmt} &\rightarrow \text{scanf} ( \text{TEXT\_STRING} , \&\text{ID} ) \text{;} \\
    \text{IdList} &\rightarrow \text{ID} \mid \text{ID} , \text{IdList} \\
    \text{UnaryStmt} &\rightarrow \text{IncDecOp} \ \text{ID} \text{;} \mid \text{ID} \ \text{IncDecOp} \text{;} \\
    \text{IncDecOp} &\rightarrow \text{++} \mid \text{--} \\
    \text{AssignExprOp} &\rightarrow \text{=} \mid \text{+=} \mid \text{-=} \mid \text{/=} \\
    \text{MathExpr} &\rightarrow \text{MathOperand} \mid \text{MathOperand} \ \text{MathOp} \ \text{MathExpr} \\
    \text{MathOperand} &\rightarrow \text{ID} \mid \text{VALUE\_NUM} \\
    \text{MathOp} &\rightarrow + \mid - \mid * \mid / \mid \% \\
    \text{Cond} &\rightarrow \text{CondAnd} \ \text{CondOrPrime} \\
    \text{CondOrPrime} &\rightarrow || \ \text{CondAnd} \ \text{CondOrPrime} \mid \varepsilon \\
    \text{CondAnd} &\rightarrow \text{CondRel} \ \text{CondAndPrime} \\
    \text{CondAndPrime} &\rightarrow \&\& \ \text{CondRel} \ \text{CondAndPrime} \mid \varepsilon \\
    \text{CondRel} &\rightarrow \text{MathExpr} \ \text{RelOp} \ \text{MathExpr} \mid \text{CondUnary} \\
    \text{CondUnary} &\rightarrow \text{!} \ \text{CondUnary} \mid (\text{Cond}) \mid \text{IdVal} \\
    \text{IdVal} &\rightarrow \text{ID} \mid \text{VALUE\_NUM} \mid \text{VALUE\_CHAR} \\
    \text{RelOp} &\rightarrow == \mid != \mid < \mid > \mid <= \mid >=
\end{align*}
\end{small}

\newpage

\textbf{4. Condiciones Lógicas (Precedencia por Capas):} 

Para respetar rigurosamente la jerarquía de las operaciones lógicas, las reglas de producción para las condiciones (\texttt{Cond}) se diseñaron utilizando un enfoque de niveles de prioridad estratificados:

\begin{itemize}
    \item \textbf{Nivel de Menor Prioridad (OR):} El análisis inicia en \texttt{Cond}, obligando a evaluar primero las capas inferiores antes de procesar el operador \texttt{||}.
    \item \textbf{Nivel Intermedio (AND):} A través de \texttt{CondAnd}, los operadores agrupan sus operandos lógicos antes de que la capa superior (OR) pueda procesarlos.
    \item \textbf{Nivel de Comparaciones (Relacional):} La regla \texttt{CondRel} evalúa las expresiones matemáticas separadas por operadores relacionales ($>$, $<$, $==$).
    \item \textbf{Nivel de Máxima Prioridad (NOT y Paréntesis):} La regla \texttt{CondUnary} asegura que las negaciones lógicas y las expresiones anidadas entre paréntesis se resuelvan primero.
\end{itemize}

\textbf{Descripción formal del autómata:}

\vspace{0.3cm}

El autómata finito determinista se define formalmente como la 5-tupla $M = (Q, \Sigma, \delta, q_0, F)$ donde:

\begin{itemize}
    \item $Q = \{q_0, q_1, q_2, \ldots, q_{46}\}$ \quad (47 estados)
    \item $\Sigma$ es el conjunto de símbolos de entrada (terminales y no terminales de la gramática)
    \item $\delta : Q \times \Sigma \rightarrow Q$ es la función de transición definida a continuación
    \item $q_0$ es el estado inicial
    \item $F = \{q_0\}$ es el conjunto de estados de aceptación
\end{itemize}

\textbf{Observaciones sobre el autómata:}

\begin{enumerate}
    \item \textbf{Estado $q_0$ como estado inicial y de aceptación:} El estado $q_0$ cumple ambos roles porque la producción $\text{Stmt} \rightarrow \varepsilon$ permite aceptar la cadena vacía (un programa sin instrucciones). Después de reconocer completamente una instrucción, el autómata siempre regresa a $q_0$ para iniciar el reconocimiento de la siguiente.

    \item \textbf{Determinismo por token de anticipación (\textit{lookahead}):} Desde $q_0$, el primer token de la entrada determina unívocamente cuál camino seguir:
    \begin{itemize}
        \item \texttt{int}, \texttt{float}, \texttt{double} $\rightarrow$ DeclNum (estado $q_1$)
        \item \texttt{char} $\rightarrow$ DeclChar (estado $q_5$)
        \item \texttt{ID} $\rightarrow$ AssignStmt o UnaryStmt posfijo (estado $q_9$)
        \item \texttt{++}, \texttt{-{}-} $\rightarrow$ UnaryStmt prefijo (estado $q_{12}$)
        \item \texttt{if} $\rightarrow$ IfStmt (estado $q_{14}$)
        \item \texttt{while} $\rightarrow$ WhileStmt (estado $q_{20}$)
        \item \texttt{for} $\rightarrow$ ForStmt (estado $q_{26}$)
        \item \texttt{printf} $\rightarrow$ PrintStmt (estado $q_{36}$)
        \item \texttt{scanf} $\rightarrow$ ScanStmt (estado $q_{40}$)
    \end{itemize}

    \item \textbf{Transiciones opcionales ($\varepsilon$):} Los estados $q_2$ y $q_6$ poseen dos transiciones por \texttt{;}: una directa a $q_0$ (cuando la variable se declara sin inicializar, es decir $\text{AssignOpNum} \rightarrow \varepsilon$) y otra que pasa por la asignación antes de llegar al \texttt{;} final.

    \item \textbf{Ambigüedad resuelta en $q_9$:} Cuando el primer token es \texttt{ID}, el autómata transita a $q_9$. Desde ahí, el segundo token resuelve la ambigüedad:
    \begin{itemize}
        \item Si es un \textit{AssignExprOp} (\texttt{=}, \texttt{+=}, \texttt{-=}, \texttt{/=}) $\rightarrow$ se sigue el camino de AssignStmt.
        \item Si es un \textit{IncDecOp} (\texttt{++}, \texttt{-{}-}) $\rightarrow$ se sigue el camino de UnaryStmt posfijo.
    \end{itemize}

    \item \textbf{Símbolos no terminales como transiciones:} Las etiquetas como \textit{Cond}, \textit{StmtList}, \textit{MathExpr}, \textit{PrintArgs}, \textit{ForInit}, \textit{ForCond} y \textit{ForStep} representan llamadas recursivas al parser predictivo, donde cada no terminal se expande según sus propias reglas de producción definidas en la gramática.
\end{enumerate}

\subsection{Entrada/Salida y Ejecución}

El analizador léxico desarrollado soporta dos modos de operación para entradas en lenguaje C, lectura desde archivo y lectura desde teclado.

\begin{enumerate}
    \item \textbf{Lectura desde archivo.}  
    En este modo, el usuario debe colocar el archivo en la carpeta inputs y el sistema lo lee usando codificación UTF-8 y posteriormente procesa el contenido línea por línea y al final del análisis léxico, los resultados pueden ser exportados hacia la carpeta outputs.

    \item \textbf{Lectura desde teclado.}  
    El analizador permite ingresar código fuente directamente desde la terminal. La captura finaliza cuando el usuario introduce el símbolo terminal \texttt{\$}, el cual actúa como delimitador de fin de entrada.  
 \end{enumerate}

Durante la ejecución, el sistema realiza las siguientes operaciones principales:

\begin{itemize}
    \item Tokenización del código fuente mediante el lexer de C proporcionado por la biblioteca \textit{Pygments}.
    \item Clasificación de cada token conforme a las categorías definidas en el curso (Keyword, Identifier, Operator, Punctuator, Constant y Literal).
    \item Eliminación de tokens irrelevantes como espacios en blanco o saltos de línea que no aportan información sintáctica.
    \item Agrupación y conteo total de tokens por categoría.
    \item Visualización opcional del código fuente con resaltado de tokens en colores y exportación de resultados a un archivo de texto.
\end{itemize}

Cabe mencionar que el caso particular del carácter \texttt{\&} utilizado en la función \texttt{scanf()}, así como ciertos delimitadores como paréntesis, llaves, comas y punto y coma se trataron explícitamente como punctuators debido a que la biblioteca utilizada no los categoriza de forma consistente.

\subsection{Relación con la CFG propuesta}

El diseño del analizador léxico se realizó tomando como referencia la gramática libre de contexto definida para el lenguaje C reducido considerado en este proyecto.  
Nuestra CFG contempla instrucciones como declaraciones de variables numéricas y de carácter, asignaciones, estructuras de control (\texttt{if}, \texttt{while}, \texttt{for}), operaciones unarias de incremento y decremento, así como llamadas a funciones de entrada/salida (\texttt{printf} y \texttt{scanf}) \textit{\cite{cfglexer}}.

También se definieron reglas de producción para expresiones matemáticas y condiciones lógicas respetando la precedencia de operadores mediante un enfoque de derivación por niveles, donde los operadores lógicos OR presentan la menor prioridad, seguidos por AND, operadores relacionales y finalmente operadores unarios como la negación lógica y el uso de paréntesis.

\subsection{Pruebas realizadas}
Para validar nuestro analizador léxico, se propusieron un par de pruebas utilizando fragmentos de código en lenguaje C que incluyen declaraciones, expresiones aritméticas, llamadas a funciones de entrada/salida y estructuras de control contempladas en la CFG propuesta.
\begin{itemize}
    \item \textbf{Código 1: Declaraciones y expresiones aritméticas}  

    En esta prueba se analizó un programa que incluye declaraciones de variables numéricas, una asignación con expresión aritmética y una llamada a la función \texttt{printf()}. El analizador léxico identificó correctamente palabras clave, identificadores, operadores de asignación y operadores aritméticos, así como constantes numéricas y delimitadores del lenguaje.

    De igual forma, se verificó el reconocimiento adecuado del literal de cadena utilizado en la función de salida, el cual fue agrupado como un único token. El resultado del análisis permitió obtener el conteo total de tokens clasificados por categoría, validando la correcta tokenización de expresiones aritméticas y sentencias de impresión.

    \item \textbf{Código 2: Entrada de datos y estructura iterativa}  

    En esta prueba se evaluó un programa que realiza lectura de datos mediante \texttt{scanf()} y emplea una estructura iterativa \texttt{for}. Durante el análisis se reconocieron correctamente las palabras clave asociadas al ciclo, los operadores relacionales y de incremento, así como los delimitadores correspondientes.

    También se verificó la clasificación del símbolo especial \texttt{\&} utilizado para la lectura por referencia y la correcta identificación del literal de cadena dentro de la función \texttt{printf()}. Los resultados confirmaron que el analizador puede procesar adecuadamente instrucciones iterativas y llamadas a funciones de entrada/salida, generando la tabla y el conteo agrupado de tokens.
\end{itemize}
